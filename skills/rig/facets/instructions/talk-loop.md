# instruction: talk-loop

話し言葉の発話を受け取り、意図を見極めて適切な rig コマンド（dev / sales …）へ橋渡しし、会話を継続する。talk は engine の**前段（自然言語→構造化された rig 起動）**と**会話の継続**だけを担う。実作業は委譲先と engine が行う。

## 手順（1ターン）

### ① 意図の見極め

発話が **(a) 雑談・質問** か **(b) rig アクション要求**（レビュー/実装/PR/マージ/商談レビュー等）かを判定する。

- (a) なら `talk-assistant` の語り口で短く答え、⑥へ。
- (b) なら ② へ。

### ② 正規化

発話からフィラー（あー/えーと）・言い淀み・重複・口語崩れを落とし、要点（やりたいこと・対象・条件）を抽出する。

### ③ ルーティング（動的列挙）

利用可能な `/rig:*` コマンドを**動的に列挙**し、各コマンドの description に照らして、発話の意図を最も合うドメインへ対応づける。

- コマンドの候補は固定で持たない。`commands/` 配下や現在のセッションで使える `/rig:*` を見て判断する（例: `/rig:dev` `/rig:sales`）。**新しい pack が増えれば自動的に候補に入る**。
- 「コードのレビュー/実装/PR」系 → `/rig:dev`。「商談/案件の振り返り」系 → `/rig:sales`。判別がつかなければ ⑤ で1問確認する。

### ④ 引数の抽出

対応づけたコマンド向けに、発話から flag / recipe / 対象を組み立てる。

- 例: 「今の変更だけレビューして」→ `/rig:dev --only review`。「この商談ファイルを見て」→ `/rig:sales <パス>`。「まず構成だけ見せて」→ `--plan`。
- 対象（ファイル・Issue・商談記録）が発話に無ければ**捏造しない**。⑤ で訊く。

### ⑤ 解釈の確認（確認ガード）

実行内容を**一言で宣言**してから動く。確認の強さはリスクで分ける。

| アクション | 確認 |
|---|---|
| 情報取得・review・`--plan`・状態確認など低リスク | 「〜しますね」と宣言して即実行してよい |
| 書き込み・実装・push・merge・capture など影響あり | **必ず確認を取ってから実行**（「〜して push まで行きますが、いい?」） |
| 意図・対象が曖昧／確信度が低い | **1問だけ**訊いて確定させる（質問攻めにしない） |

### ⑥ 委譲と継続

- (b) の場合、確定した `/rig:*` 起動文字列を該当コマンド経由で通常 engine（PARSE→RESOLVE→COMPOSE→RUN）へ渡す。**talk は engine 規則（context-minimal 等）を再定義せず、委譲先にそのまま従わせる**。
- 実行後は結果を**短い話し言葉で報告**（1〜2文）し、次の用件を受ける。
- 「もういい / exit / やめて」等で会話モードを終了する。

## 原則

- talk 自身は重い処理を抱えない（長い diff・ログ・ファイル全文を会話 context に引き込まない）。重い処理は委譲先と engine が context-minimal で回す。
- engine（SKILL.md）と dev / sales のフローは変更しない。talk は薄い前段である。
