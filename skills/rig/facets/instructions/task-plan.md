# instruction: task-plan

依頼を**細粒度の検証可能タスクに割ってから実装する** routing。分解の作法は委譲先 persona（`planner`）と出力 `task-plan` が持つのでここには再掲しない（Native-first）。「いきなり大きく実装」を避け、「小さく割って・確かめながら・順に潰す」に変える。

**スコープ**: 依頼 → 細粒度タスク計画（各タスクに検証つき）→ 上から実行（実装→検証→レビュー）。`/rig:goal`（反応的に次の一手）の前段＝**事前に全タスクを見渡す計画**。

## 手順

### ① 分解（plan）

`planner` を合成して、依頼を `task-plan`（タスク表＋未確定）に分解させる。長い調査は subagent へ（context-minimal）。

- **各タスクは細粒度・検証つき**（数分・少数ファイル・「どうなったら完了か」をコマンド/テスト/観察で）。
- **未確定/要調査は先に出す**（捏造でタスクを埋めない）。仕様の穴があれば、それを潰すタスク or ユーザーへの確認を先頭に。
- 既定は **interactive**：計画を提示して**承認を取ってから**実行に移る（`--plan` なら計画提示で停止）。大きく実装してから「違った」を防ぐ。

### ② 実行（上から潰す）

承認された計画を、**タスク順（依存順）に実装する**。各タスクは：

- **1タスク＝1 subagent**（context-minimal）。触るファイル・手順を渡し、実装させる。`--tdd` 指定時は red-green-refactor。
- **タスクの検証を実行**：計画の「検証」列（テスト/コマンド/grep/観察）で**そのタスクが完了したか確かめてから次へ**。未達なら次に進めない。
- 独立タスク（依存 `—`）は、`--orchestrate` の DAG 並列（`needs:`）に渡せば同時実行できる。

### ③ レビュー（2段目）

実装が一巡したら、`parallel-review`（security/design/test の並列レビュー）を `review-gate` で通す。実装時の自己検証が1段目、独立レビューが2段目。

### ④ 収束

`acceptance-gate` で「全タスクが検証済み・レビューに REJECT なし」へ収束。未達は指摘反映で再走、最大 K 回でユーザーへ。

## ガード

- **大きく曖昧なタスクを作らない／検証の無いタスクを作らない**（planner の規律）。
- **計画は承認を取ってから実行**（interactive 既定）。実装してから方針転換、を避ける。
- **未確定を捏造で埋めない**（要調査に出して先に潰す）。
- 実装・検証・レビューは委譲先と subagent が回す（context-minimal）。計画は engine の再定義ではなく、dev フローの前段。
- 配線・PR・マージまで要るなら `/rig:dev`（release-flow）へ接続する（task-plan は計画→実装→検証→レビューまで）。
