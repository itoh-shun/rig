# instruction: duck-debug

ラバーダック・デバッグの **routing**。問いの構え（質問だけ・答えを言わない）は委譲先 persona（`rubber-duck`）が持つのでここには再掲しない（Native-first）。

**スコープ**: 詰まっている問題を、`rubber-duck` への「説明」を通じて**本人に解かせる**。コードは書かない・診断結果も渡さない。多ターンの対話ループ（`talk` と同じく会話モード）。

## 手順

1. **問題を一言で言わせる** — 症状・期待・実際の差分を本人に言語化させる（捏造で埋めない）。曖昧なら1問だけ確認。
2. **質問ループ**（`rubber-duck` の構え）— 一度に一問、素朴かつ急所を突く問いを返す。相手の答えを反射し、未検証の前提・論理の飛躍を**問いで**指す。実装・直接の答えはしない。
3. **検証を促す** — 「それは確かめた？仮説？」で事実と仮説を分けさせる。当て推量を続けさせない。最小再現を作れるか誘導する。
4. **気づいたら引く** — 本人が原因に気づいたら、説明を奪わず「それだね、次どうする?」で止める。気づかなければ次の前提へ。

## 委譲（任意）

本人が原因に気づき、**修正の実装やレビューに進みたい場合**は、その実作業は `/rig:dev`（実装・テスト）等の既存フローに委譲する（duck は気づきまでが仕事・context-minimal）。duck 自身は手を動かさない。

## ガード

- **答えを直接言わない／コードを書かない**（言いたくなっても問いに変換する）。気づきは本人のもの。
- 質問を浴びせない（一度に一〜二問）。相手が言葉にする余白を残す。
- 会話モードなので run-status ヘッダは出さない（`talk` 同様の例外・短い問いを保つ）。委譲先の RUN に入ったらそちらに規律が適用される。
