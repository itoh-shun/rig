# instruction: intake

依頼の「何を／なぜ／どこ／どこまで」を確定し、作業サイズを見積もる。

## 手順

### ① スコープ確認

以下の4点を確認する。まだ決まっていない項目があれば、次のステップに進む前にユーザーに問い直す。

- **何を**: 変更・追加・修正の対象（機能、画面、API など）
- **なぜ**: ビジネス上の理由・課題
- **どこ**: 影響範囲（対象ファイル、レイヤー、サービス）
- **どこまで**: 完了条件（AC）

スコープが曖昧・広すぎる場合は `superpowers:brainstorming` スキルを参照し、ユーザーとともに範囲を絞り込む。

**「どこ」の特定にコードベース探索が要る場合（§8 Native-first）**：ホストが read-only の **Explore エージェント型**を公開していれば、探索 dispatch はそれで行う——構造的に書き込み不能なので調査段階の dispatch として安全で、汎用 subagent より速く安い。無ければ通常の subagent で探索する（従来通り）。親 context に探索結果の生ログを引き込まない原則（context-minimal）はどちらでも同じ。

### ② サイズ見積もり

| サイズ | 目安 |
|--------|------|
| S | 1〜2ファイル、既存パターンの踏襲、半日以内 |
| M | 複数ファイル・レイヤー跨ぎ、新設ロジックあり、1〜2日 |
| L | アーキテクチャ変更・外部依存追加、設計フェーズ必須 |

### ③ Issue 起票の判断

Issue 起票の要否は recipe または呼び出し時フラグ（`--issue` など）に従う。manifest にデフォルトが定義されている場合はそれを優先する。

### ④ 次ステップへの引き継ぎ

確定した「何を／なぜ／どこ／どこまで」とサイズ（S/M/L）を後続ステップへ渡す。
