# instruction: identify-behavior-boundaries

**refactor 専用の安全境界確認。** 内部構造を変える前に「変えてはいけない外部から見た挙動」を明文化する。これが無いまま refactor に入ると「動いているように見えるが実は挙動が変わった」を見逃す。

## 手順

### ① 観測可能な契約の列挙

対象コードについて、以下を洗い出す：

- **公開インターフェース**（関数シグネチャ・公開クラス・エクスポートされる型）
- **副作用**（I/O・DB 書き込み・外部 API 呼び出し・ログ出力）
- **エラー時の挙動**（例外の型・エラーメッセージ・リトライ有無）
- **性能特性**（明示的な SLA/タイムアウトがあれば）

### ② 変えてよい境界・変えてはいけない境界の分離

上記のうち、依頼された refactor の目的（責務分離・重複除去・可読性向上）に**無関係なもの**は「変えてはいけない境界」として固定する。目的上どうしても変える必要があるものは「意図的な変更」として明示し、後続の `review_diff` step で reviewer に伝える。

### ③ 引き継ぎ

固定した境界のリストを `implement` step の入力として渡す。`compare_behavior` step（実装後）はこのリストに対して差分がないことを確認する（境界リストが answer key になる）。
