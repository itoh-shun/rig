# instruction: implement

実装を行う。ビジネスリスクとテストコストを比較し、TDD か直接実装かを選択する。

## 手順

### ① 実装方針の選択

| 条件 | 方針 |
|------|------|
| ビジネスリスク ＞ テストコスト（コアロジック、副作用、境界値） | `tdd` スキルを起動してレッド→グリーン→リファクタのサイクルで実装する |
| それ以外（UI 微修正、設定変更、スキャフォールドなど） | `tdd` を省略して直接実装する |

### ② subagent への委譲

実装はコンテキストを最小化した subagent 上で行う。subagent に渡す情報：

- intake で確定したスコープ・AC
- design フェーズがあれば設計ドキュメント
- 選択した実装方針（TDD / 直接）

subagent の実装が完了したら diff をレビューしてから verify へ進む。

**大きめの実装でのストリーミングgate（opt-in・#302）**：実装が長時間・大規模になる見込みの場合（`--diff-git`測定でsize=L/XL相当、または複数ファイルにまたがる実装）、subagentは実装の節目（意味のある単位で1ファイル書き終えた時点等）で軽量チェック（型チェック・lintのみ。フルテストは含めない）を都度実行してよい。目的は「verifyまで待って大量の指摘をまとめて食らう」手戻りを減らすこと。ここでの指摘はブロッキングにせず実装への即時フィードバックとして使い、最終的な合否は通常通り②verify以降のacceptance-gateで決める。小さい変更ではこのオーバーヘッドは不要（都度チェックを省いてよい）。

### ③ TDD サイクル（tdd スキル委譲）

`tdd` スキルを使う場合、以下を subagent 経由で `tdd` に委譲する：

1. 失敗するテストを先に書く（Red）
2. テストが通る最小実装を書く（Green）
3. 重複・複雑さを取り除く（Refactor）

`tdd` スキルの詳細手順は当該スキルに委ねる。本ファイルで再実装しない。
