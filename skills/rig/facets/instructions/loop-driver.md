# instruction: loop-driver

対象を**一定間隔 or 自己ペースで繰り返す** recurring ループの routing。新しい制御機構は発明しない。**`patterns/autonomous-loop`（`ScheduleWakeup` による次 tick 予約）を再利用する**だけ。各 tick の実作業は委譲先（`/rig:*` フローや任意コマンド）と subagent が回す。loop は周回の駆動と停止判定に徹する（context-minimal）。

> goal（達成まで収束）との違い：loop は**同じ対象を繰り返す**。終端は「達成基準」ではなく「**停止条件・回数・明示停止**」。終わりのない見張り・ポーリング・定期実行に使う。

## 手順

### ① 対象・間隔・停止条件の確定（最初の1回）

- **対象**：繰り返す 1 アクション。`/rig:dev`/`/rig:pr` 等の rig フロー、または任意コマンド（CI 確認・集計等）。
- **間隔**：`--every <dur>`（時間駆動・例 `10m`）／指定が無ければ**自己ペース**（イベントやユーザー合図で次へ）。
- **停止条件（必須・いずれか）**：
  - `--until "<check>"`：機械検証（shell が exit 0／GitHub MCP の status／grep）で「終わったか」を判定。例「CI が green」「対象 Issue がクローズ可能」。
  - `--times <N>`：N 回で終了。
  - 明示停止：ユーザーが止めるまで（**この場合は安全上限を確認**してから回す＝無限監視に黙って入らない）。
- **曖昧・停止条件も上限も無いなら 1 問だけ確認**して確定する（暴走防止）。`--plan` なら対象・間隔・停止条件を `### Loop Config: ループ設定` ブロック（正本: `facets/instructions/plan`）の正準フォーマットで提示して停止。

### ② 1 tick 実行

確定した対象を委譲実行する（rig フローなら通常 engine：PARSE→RESOLVE→COMPOSE→RUN へ。任意コマンドはそのまま実行）。**長い出力は親に引き込まず**、判定に要る要点だけ受け取る（context-minimal）。

### ③ 停止判定

- `--until` → 機械/観察で停止条件を照合（満たせば終了）。`--times` → カウントを進める。
- **進捗のある監視か**を見る：毎 tick 同じ結果なら「待ち（pending）」として継続、変化があれば対応（例 CI 失敗→`/rig:dev` で修正を委譲）。

### ④ 次 tick 予約 / 終了

| 判定 | 動作 |
|---|---|
| **停止条件を充足 / N 回到達** | 終了。何を確認/実行したかを**短く**報告。 |
| **継続（時間駆動）** | `patterns/autonomous-loop` の `ScheduleWakeup` で次 tick を予約（`delaySeconds`：キャッシュ温 270／冷 1200+／**300 禁忌**）。`prompt` は次 tick を再開できる正準状態（対象・間隔・停止条件・経過 tick）を含める。 |
| **継続（自己ペース）** | 次の合図/イベントを待つ。 |
| **安全上限到達 / 進捗ゼロが続く** | **停止してユーザーへ**。何が起きているか・選択肢を提示（無限ループ禁止）。 |

## ガード

- **停止条件 or 上限が無いまま回さない**（`--until`/`--times`/明示合意のいずれか）。暴走防止。
- **各 tick を報告**する（沈黙で回り続けない）。tick ログは簡潔に（毎回同じ run-status を全文再掲しない）。
- 時間駆動は `ScheduleWakeup` 規約（270/1200・300 禁忌）を守る。`sleep` でブロックしない。
- 書込/push/merge を伴う対象は tick ごとに委譲先の step ゲートで確認（`--autonomous` でも capture ゲートは解除されない）。
- loop は周回ドライバであって engine の再定義ではない。各 tick の中身は委譲先が回す（Thin Harness, Fat Skills）。
