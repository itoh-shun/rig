# instruction: test-design

テストケース設計の **routing**。設計の作法（7観点・正直さ・トラック分化）は委譲先 persona（`test-designer`）と knowledge（`qa-test-lenses`）が持つのでここには再掲しない（Native-first）。対象は機能・課題（PBI）・diff・仕様・現行挙動。テストの**実行**は担当外（設計まで）。

**スコープ**: AI 生成テストの定番の穴＝**正常系偏り（観点抜け）**と**新機能/移行で軸がぶれる**を、固定の7観点とトラック分化で塞ぐ。納品はケース表＋カバレッジ要約＋仕様ギャップ。

## 手順

1. **対象とトラックの確定** — 対象を1つに確定（曖昧なら1問だけ確認）。**トラックを決める**：既定は**新機能**（仕様の起点＝課題＋ソースコード／問い＝要件を満たすか）。`--migration` 指定時は**移行**（起点＝現行ヘルプ・現行挙動・デザイン／問い＝現行どおり動くか）。長文・大量コードは親 context に引き込まない（subagent へ・context-minimal）。
2. **knowledge 注入必須** — `knowledge/qa-test-lenses` を Knowledge 位置に注入して `test-designer` を合成（agent が無ければ `facets/personas/test-designer`）。
3. **7観点の一巡** — L1 初見ユーザー／L2 現場ベテラン／L3 悪意ある操作者／L4 データ整合性監査役／L5 移行担当者／L6 回帰デグレ番人／L7 仕様懐疑者 を必ず一巡。**各観点が最低1ケース**を残す（出ない観点は「該当なし＋理由」）。正常系（L1）を1つ押さえたら重心を L2〜L7 へ。大規模・重要機能では観点ごとに `patterns/parallel-fanout` で並列に出してもよい。
4. **根拠と正直さ** — 各ケースに **Test Basis（一次情報）** を紐づける。コード未確認・仕様未確定は期待結果を断定せず `※要確認（未実施）` と書く。各ケースを**テスト可／保留／不可**に分類し、不可・保留は**仕様ギャップ（差し戻し）**に再掲する（黙って埋めない＝捏造で「テスト可」にしない）。
5. **タグ付け** — ISO/IEC 25010 の品質特性タグと、工数（1・3・5・8）を付ける。
6. **機密スクリーニング** — 出力前に顧客名・メール・電話番号・本番データの混入を検知し、ダミー/伏字へ。外部共有物に実データを出さない。
7. **出力** — `output-contracts/test-cases`（観点カバレッジ行＋ケース総数＋ケース表＋仕様ギャップ＋移行時は要件カバレッジ表）。

## `--review` モード（既存ケースのレビュー・指摘のみ）

既存のテストケース（CSV / 表 / 文書）を渡された場合は、**設計せず指摘に徹する**：7観点で抜けを洗い、根拠の欠落・正常系偏り・未確認の断定・カバレッジ分類の誤りを**指摘リスト**で返す。**勝手に修正しない**（修正は人間が採否を判断してから）。出力は `output-contracts/review-verdict`（PASS/REJECT＋指摘）に観点カバレッジを併記。

## ガード

- **正常系に偏らない**・**7観点を取りこぼさない**（各 ≥1 か「該当なし＋理由」）。網羅した風の水増しもしない。
- **根拠の無い期待結果を書かない**／**未確認を断定しない**（`※要確認（未実施）`）。
- **実行・合否・修正をしない**（AI はテスト設計者であってテスターでない）。
- **テスト不可・保留は仕様ギャップに名指す**（仕様への差し戻し）。
- **機密を素通ししない**。正本は消さず `DEPRECATED` 印（履歴保持）。
