# instruction: brainstorm

ラフな着想を**質問→代替案→セクション合意で固める**壁打ちの routing。発散と収束の作法は委譲先 persona（`brainstormer`）と出力 `design-brief` が持つのでここには再掲しない（Native-first）。実装やタスク分解の**前段**＝「何を作るか／なぜ／どの順か」を先に固め、「曖昧なまま実装に突っ込む」を防ぐ。

**スコープ**: ぼんやりした要望 → 質問で詰める → 2〜3案を出す → 設計をセクションで合意 → `design-brief` に収束 → 次段（`/rig:tasks`/`/rig:dev`）へ。

## 手順

### ① 発散（質問で詰める）

`brainstormer` を合成し、**決め打ちせず質問**から入る。不明点・前提・制約・成功条件を確認する。曖昧さがあれば、まずそれを名指す（当て推量で設計を進めない）。

### ② 代替案

**2〜3の代替案**を出し、トレードオフを並べる。1案だけで進めない。そのうえで推しを1つに収束させる（根拠つき）。

### ③ セクション合意（interactive）

設計を**節（データ / UI / 失敗時 / 移行 / 範囲 等）に分け、1つずつ提示して承認/修正を取る**。一気に全部決めない。合意の取れない節は「未解決」に回す。**承認を取りながら進む**のが既定（`--plan` なら brief 草案を提示して停止）。

### ④ 収束

`design-brief`（固めた狙い／セクション別の決定〔根拠つき〕／検討した代替案／未解決の問い／次の一手）にまとめる。未解決は隠さず出す。

### ⑤ 接続

承認された brief を次段へ渡す：細粒度に割るなら `/rig:tasks`、そのまま実装なら `/rig:dev`。未解決が重いなら調査を先に。

## ガード

- **決め打ちしない／代替を必ず見る／セクションで合意する**（brainstormer の規律）。最初の案に飛びつかない。
- **承認を取りながら進む**（interactive）。一気に設計を確定して「違った」を避ける。
- **未解決を捏造で埋めない**（要調査に出す）。確認していない制約・既存実装を断定しない。
- **実装に踏み込まない**：何を・なぜ・どの順、まで。コード/ファイル段取りは次段（`/rig:tasks`）。
- 壁打ちは dev フローの**前段**＝engine の再定義ではない。長い調査は subagent へ（context-minimal）。
