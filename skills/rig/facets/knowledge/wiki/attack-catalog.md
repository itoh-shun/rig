`exploit-researcher` / `threat-modeler` が **能動的にコードを攻める**ときの技法カタログ（事実）。`appsec-checklist` が「diff のどこに何が潜むか（守り）」を並べるのに対し、ここは「実際にどう突くか＝PoC の形」を並べる。判断（本当に刺さるか）は persona が持つ。**捏造禁止**：ここにある技法は「試す観点」であって「存在の証明」ではない。刺さると示せて初めて所見になる。

## スコープと倫理境界（最重要）

- **対象は自プロダクトのコードと、明示的に検証を許可されたローカル/ステージング環境のみ。** 本カタログの技法を第三者のシステム・本番・スコープ外資産に向けてはならない。
- **このハーネスは静的解析＋ローカル検証まで。** 動いている外部サービスへ攻撃トラフィックを送る動的スキャン（DAST）は既定で範囲外。実施する場合は `.rig/security-targets.json` の allowlist で明示許可したホストに限る（未設定なら実行しない）。
- ここに書くのは**確認手順**であって、破壊・持ち出し・永続化の手口ではない。PoC は「刺さることを最小限で示す」ところで止める（例: `id` を撃つのでなく「任意コマンドが argv に混入する経路がある」ことを示す）。

## 入力起点の突き方（境界を越えられるかを試す）

- **SQL/NoSQL** — 文字列連結クエリに `' OR 1=1 --`／エスケープ後も残る二次注入（stored → 別クエリで展開）。**ブラックリスト回避**：`OR`→`oR`、コメント挿入 `UN/**/ION`、引用符除去実装なら引用符不要な数値文脈を狙う。canonical はパラメタライズ（プレースホルダ＋値分離）。
- **コマンド注入** — `shell=True`/バッククォート/`exec` 系へ外部入力。デニーリスト（`;`&`|`）は **`$(...)` コマンド置換・改行・グロブ**で回避しうる。canonical は argv 配列化＋`shell=False`。
- **パス・トラバーサル** — `../` 正規化漏れに加え、**絶対パス**（`os.path.join(base, "/etc/passwd")` は base を捨てる）・zip-slip・シンボリックリンク・URL エンコード `%2e%2e`。canonical は realpath 後に base 配下かを検査。
- **SSRF** — 内部アドレスのブロックを**別表記で回避**：`169.254.169.254`（クラウドメタデータ）・10進/16進 IP（`2130706433`/`0x7f000001`）・`[::1]`・DNS リバインディング・リダイレクト追従。canonical は解決後 IP を `ipaddress` で private/loopback/link-local 判定。
- **デシリアライズ** — pickle/yaml.load/ObjectInputStream に外部データ → `__reduce__` 等で任意実行。canonical は安全形式（JSON）＋スキーマ検証。
- **XXE / SSTI / XSS** — 外部エンティティ展開・テンプレート式評価・`innerHTML`/`javascript:` スキーム。

## 認可・認証の突き方（「分岐の欠如」を突く）

- **IDOR** — 自分のトークンで他人の ID を直指定（`GET /doc/2`）。一覧では絞れていても単体 GET/PUT で漏れる型が定番。「認証済み＝認可済み」と取り違えた実装（ログインだけ確認しオーナー検査なし）を狙う。
- **権限昇格** — 認可が UI 側のみで API 側に無い／管理エンドポイントのミドルウェア外し忘れ／mass-assignment で `role` を送り込む。
- **トークン/セッション** — JWT `alg:none`・署名未検証・`alg` 混同（HS/RS）・失効なし・予測可能なリセットトークン（`random` は再シードで再現可能＝CSPRNG が必要）。

## 暗号・秘密の突き方

- **弱いパスワード保存** — MD5/SHA-1、ソルト無し（同一パスワードが同一ハッシュ＝レインボーテーブル・共有検知）。canonical は per-password ソルト＋低速 KDF（bcrypt/scrypt/argon2/PBKDF2）。
- **予測可能な乱数** — `random`/`Math.random` をトークン・リセット・セッションに使用。再シードで再現できれば予測可能。
- **ハードコード秘密** — 鍵/接続文字列/トークンのリテラル埋め込み・コミット履歴残留（削除コミットでは消えない＝ローテーション必須）。

## 確認の型（所見にする前に必ず）

1. **入力元 → 危険な操作**までのデータフローを一本の線で言えるか（信頼境界を1つ以上越えるか）。
2. **「誰が・何を送ると・何が起きる」**を1行で書けるか。書けない＝所見にしない。
3. 最小 PoC（入力値・期待される異常挙動）を示せるか。示せない懸念は **`情報不足`** として分離し、`Critical` を付けない。
4. 既存の緩和（WAF・フレームワークの自動エスケープ・認可ミドルウェア）で実際に止まらないかを確認したか。
