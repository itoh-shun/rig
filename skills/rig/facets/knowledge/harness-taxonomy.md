# knowledge: harness-taxonomy

プロジェクトの**ハーネス（エージェントで開発する仕組み全体）を点検**するときの観点カタログ（事実）。判断は `personas/harness-auditor` が持つ。ここは「ハーネス要素を何で分類し・どこが穴になりやすいか」だけを並べる。`knowledge/harness-engineering`（brick 合成の作法）とは別物＝こちらは**自分のハーネスの健全性監査**。

> ハーネスには**2層**ある：**エージェントハーネス**（プロダクト内蔵＝Claude Code/Codex 側のループ・ツール・メモリ・コンテキスト管理）と、**ユーザーハーネス**（使う側が組む＝CLAUDE.md/Skills/Hooks/MCP・テスト・lint・CI・rig の recipe）。rig が作り、ここで監査するのは**ユーザーハーネス**。`ハーネス ⊇ コンテキスト ⊇ プロンプト` の包含関係。

## 2×2 フレームワーク（ハーネス要素はこの4象限に分類できる）

2軸で分ける。**① 計算的（決定論的・コードで判定）か / 推論的（LLM の判断）か** ／ **② ガイド（先回りで方向づける＝フィードフォワード）か / センサー（事後に検知する＝フィードバック）か**。

| | **ガイド**（先回り・方向づけ） | **センサー**（事後・検知） |
|---|---|---|
| **計算的**（決定論的） | LSP・型・CLI・コードモッド・scaffold・テンプレ | **lint・型チェック・テスト・build・ArchUnit・CI** |
| **推論的**（LLM 判断） | AGENTS.md/CLAUDE.md・Skills・設計ドキュメント・persona | **AI コードレビュー・LLM-as-judge・review-gate** |

- rig のブリックを当てると：persona＝**推論的センサー**、instruction/knowledge＝**推論的ガイド**、acceptance-gate の機械検証（build/lint/test）＝**計算的センサー**、hook/manifest/scaffold＝**計算的ガイド/強制**。
- **4象限が揃って強い**。推論だけ（レビューはするがテストが無い）も、計算だけ（テストは通るが設計の妥当性を誰も見ない）も穴。どの象限が薄いかを見る。

## 穴になりやすい所（重い順）

1. **計算的センサーがループに組み込まれていない（最頻）**。テストや lint が「存在する」だけで、エージェントの実行ループに**バックプレッシャー**として入っていない。`CLAUDE.md に「テストを実行して」と書く` ≪ `hook で毎回強制する`。**prose の依頼は守られないことがある／hook の強制は決定論的**。信頼性が段違い。
2. **検証ループの欠落**。「モデルに自分の仕事を検証する手段を与えると品質が 2〜3 倍」（Boris Cherny）。バックプレッシャーが無いと**エージェントは自己評価に甘くなる**（→ `policies/independent-verification`：採点者≠生成者）。計算的センサーは sweet-talk できない＝最強のバックプレッシャー。**計算的センサーを一次・推論的センサーを二次**に置く。
3. **評価と計測をしていない**。ルールを足したが効果を測っていない。**善意のルール追加が逆効果**になることがある（Context Rot＝文脈が長いほど性能劣化／ルール肥大が判断を濁す）。足したら**減らす候補**も持つ。
4. **ハーネスが厚すぎる**。**Thin Harness, Fat Skills**（Garry Tan）＝ループ管理は薄く（〜200行）、知能は Skills に、実行は決定論的ツールに委譲。rig の「engine 不変・pack 上乗せ・context-minimal」と同型。親ループ/親 context が肥大したら設計が間違っている。

## 監査の問い（プロジェクトのハーネスに当てる）

- **計算的センサーはあるか／ループに入っているか**：lint・型・テスト・build が存在し、かつ acceptance-gate や hook で**強制**されているか（prose 依頼で済ませていないか）。
- **推論的センサーは独立か**：レビュー/判定が生成者と別か（自己採点で甘くなっていないか）。
- **ガイドは推論・計算の両方あるか**：CLAUDE.md/Skills（推論ガイド）だけでなく、scaffold/テンプレ/CLI（計算ガイド）で先回りしているか。
- **強制は決定論的か**：重要ルールが prose 止まりでなく hook/CI で強制されているか。
- **薄いか**：親ループ/親 context が薄く、知能が Skills 側にあるか。
- **測っているか**：ルールの効果を計測しているか。肥大したルールの**削減候補**を持っているか。
