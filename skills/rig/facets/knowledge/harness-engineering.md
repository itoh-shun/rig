# knowledge: harness-engineering

COMPOSE フェーズで `orchestrator` ペルソナへ注入される汎用知識。「ブリックを組み合わせてタスクハーネスを作る」考え方の短い記述。

## ハーネスとは

**タスクハーネス** = 1つの依頼を完了させるために、その場で組み上げる使い捨ての実行計画。recipe（固定テンプレート）を土台にしつつ、manifest や実行時フラグで brick を差し替えたり追加したりして作る。

## ブリックの組み合わせ方

1. **recipe を選ぶ** — ユーザーの意図とサイズ感に最も近い recipe を出発点にする。
2. **条件ステップを解決する** — size / flag に基づき、条件付きステップを ON/OFF する。
3. **manifest で上書きする** — プロジェクト固有の instruction / policy / persona があれば recipe の brick を置換する。
4. **不足 brick を追加する** — 対応する brick が存在しない場合のみ、インラインで最小限の指示を書く（brick を新設しない）。
5. **ハーネスを提示する** — `--plan` モードではハーネス全体を人間可読な形で提示し、承認を得てから実行に移る。

## 組み合わせの原則

- **最小構成優先**: recipe の既定ステップを尊重し、必要以上に brick を足さない。
- **再利用優先**: 同じ制御ロジックを2回書きたくなったら既存 pattern brick を参照する。
- **透明性**: ハーネスのステップ列・使用パターン・適用ポリシーをユーザーへ明示する。
- **ゲート尊重**: `review-gate` など安全ゲートを無断でスキップしない。
