# knowledge: orchestration-patterns

オーケストレーター判断時に注入される汎用知識。「どの制御フローツールをいつ使うか」のマトリクスと、recipe 化の指針。

## 制御フロー選択マトリクス

| ユースケース | 推奨パターン | 理由 |
|-------------|-------------|------|
| 1回きりの並列調査・複数観点の同時収集 | `Agent × N`（直接 dispatch） | 軽量、recipe 化不要、独立したコンテキストで並列実行 |
| 実装＋設計レビュー＋テストレビュー（3段直列） | `serial` × 3ステップ | 依存関係あり、順番が意味を持つ |
| 長時間 CI 待ち・ポーリング | `autonomous-loop` または `monitor` | CI 完了を検知するまで待機、親コンテキストを汚染しない |
| 定型並列レビュー（security/design/test） | `parallel-fanout` + `review-gate` | N 観点を1メッセージで dispatch し verdict を機械集約 |
| 大規模実装・独立タスクが多数ある場合 | subagent-driven（Agent を複数）| タスクを独立分割し並列または逐次に dispatch |

## recipe 化の指針

同じパターンを **3回以上繰り返す**場合は、recipe 化を検討する。ただし重い専用ワークフローエンジン化は行わない（context-minimal 原則を守る）。

- **recipe 化してよい基準**: ステップ列が固定されており、チームが毎回同じ順序で実行する。
- **recipe 化しない基準**: 1回きりのアドホックな調査・実験的な並列実行・特定タスク固有のステップ列。
- **重厚化を避ける基準**: 重い多段 fan-out を毎回自動実行したい衝動が生じても、明示 opt-in なしには実行しない。

## 軽さ優先の原則

- 最初は直接 Agent dispatch（plain `Agent()` call）から始める。
- 複雑な制御が必要になってから pattern brick を追加する。
- パターンを増やすほど保守コストが上がるため、YAGNI を厳守する。
