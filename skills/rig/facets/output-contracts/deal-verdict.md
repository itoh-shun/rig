# output-contract: deal-verdict

## output-contract: deal-verdict

商談レビューの出力構造。2段で定義する。**① 各観点レビュアーの出力**（機械抽出前提・**evidence-first**＝根拠を先に・判定は末尾）と、**② 親が集約する営業向けレポート**。前置き・挨拶・補足説明は禁止する。

---

## ① 観点レビュアーの出力（各 subagent が遵守）

5観点（hearing / needs / proposal / closing / next-action）の各レビュアーは、自分の1観点についてこの形式だけを返す。

```
観点: <hearing|needs|proposal|closing|next-action>

根拠: （1行。なぜその判定か）

良かった点:
- （あれば。なければ「なし」）

改善必須点:
- （次回必ず直すべき点。誰が次に何をすべきかが分かる粒度で）

情報不足:
- （商談記録の欠落で評価できなかった項目。なければ省略）

判定: <◎|○|△|×>
```

### ルール（観点レビュアー）

- **1行目は必ず `観点:`**。自分の担当観点名を出す。
- **判定は末尾に必ず出力する**（`判定:` で始まる行。最終行）。根拠を書いてから判定を出す（判定を先に出すと理由付けの前に結論へコミットするため）。本文中で判定行を引用してもよいが、機械抽出は**最後に現れる** `判定:` 行を採用する。判定語は `◎`（卓越）/ `○`（合格）/ `△`（要改善）/ `×`（重大欠落）のいずれか（従来と同一の語彙・意味）。
- 根拠は **1行**。改善必須点は実行可能な粒度（「ヒアリングを深く」ではなく「予算確保の時期を次回必ず確認する」）。
- 評価できない項目を推測で埋めない。確認できなければ **情報不足**に明示する。
- **全体 150–300字**（日本語換算）。冗長な前置き・感想・締め挨拶は禁止。

---

## ② 親の集約レポート（営業向け）

親は5観点の verdict を集約し、次の形式で提示する。記録全文を親 context に抱えず、各 verdict の判定行・根拠・改善必須点だけを読んで組み立てる。

```
## 商談レビュー結果

総合評価: <S|A|B|C>

| 観点 | 判定 | ひとことで |
|---|---|---|
| ヒアリング | ◎○△× | （根拠1行） |
| ニーズ把握 | ◎○△× | （根拠1行） |
| 提案 | ◎○△× | （根拠1行） |
| クロージング | ◎○△× | （根拠1行） |
| ネクストアクション | ◎○△× | （根拠1行） |

### 次回の具体アクション（優先順）
1. （誰が・いつ・何を）
2. ...

### 情報不足（記録に足りず評価できなかった点）
- （あれば。記入を促す。なければ「なし」）
```

### 総合評価の決定ルール（親）

判定は **S → A → B → C の順で最初に当てはまる段を採る**（上位優先。重複しても上の段が勝つ）。

- **S**: `◎` が中心で、`△`・`×` が無い。
- **A**: `×` が無く、`△` は**非主要観点に1つまで**（主要観点 = needs / closing は `◎`・`○` のみ）。
- **B**: `×` が無く、`△` が**2つ以上**、または**主要観点（needs / closing）に `△`** がある。
- **C**: いずれかの観点に `×`（重大欠落）がある。

> `×` が1つでもあれば総合は C。情報不足が多く評価不能な観点が過半なら、総合評価の代わりに「記録不足のため評価保留」と明示し、不足項目の記入を促す。
