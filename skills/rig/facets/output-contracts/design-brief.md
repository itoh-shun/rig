# output-contract: design-brief

## output-contract: design-brief

壁打ち（`brainstormer`）の確定アウトプット。ラフな着想を**質問→代替案→セクション合意**で固めた結果を、機械抽出できる形で出す。前置き・挨拶は禁止。

### 形式

```
## rig design-brief: <お題の1行サマリ>

固めた狙い: <何を・なぜ・1〜2行>
前提・制約: <分かっている制約。無ければ「特になし」>

### 決めたこと（セクション別・承認済み）
- **<セクション名（例: データ）>**: <決定> — <根拠 / 退けた案があればその理由>
- **<セクション名（例: 失敗時）>**: <決定> — <根拠>
- …

### 検討した代替案
| 案 | 概要 | トレードオフ | 採否 |
|---|---|---|---|
| A | … | … | 採用 |
| B | … | … | 不採用（理由） |

### 未解決の問い
- <まだ決まっていない・要調査の点。無ければ「なし」>

### 次の一手（推奨を1つ・起動文字列つき）
- **推奨**: `<状況から選んだ1つ。重い未解決があれば調査、規模が大きければ /rig:tasks、小さく明確なら /rig:dev>` — <なぜそれか・1行>
  - 例: `/rig:tasks "<brief を割る対象>"`（規模大／段取りが要る）
  - 例: `/rig:dev "<実装対象>"`（小さく明確）
- **他の選択肢**: <次点を1〜2個>
- → ユーザーの合意を得てから次段へ（無断で auto-chain しない）
```

### ルール

- **固めた狙い**を先頭に1〜2行で必ず出す（機械抽出の入口）。
- **決めたことはセクション別**に並べ、各決定に**根拠（と退けた案の理由）**を添える。一気に1つの結論で済ませない。
- **代替案を最低1つ**は表に出す（検討した形跡を残す）。1案だけで「これで」は不可。
- **未解決の問いを隠さない**（捏造で埋めない）。決まっていないことは正直に出す。
- **実装の詳細は書かない**（何を・なぜ・どの順、まで）。コード/ファイル単位の段取りは「次の一手」で `/rig:tasks` へ渡す。
- 全体は簡潔に。承認は**セクション単位**で取れている前提（合意の取れていない節は「未解決」へ）。
