# output-contract: task-plan

## output-contract: task-plan

プランナー（`planner`）の出力構造。依頼を**細粒度の検証可能タスク**に割った計画を、機械抽出できる固定フォーマットで出す。前置き・挨拶は禁止。

### 形式

```
## rig task-plan: <対象の1行サマリ>

概要: <何を作る/直すか・1〜2行>
未確定/要調査: <仕様の穴・前提不明・調査が要る点を箇条書き。無ければ「なし」>
タスク数: <N>（うち並列可: <M>）

### タスク
| # | 目的（1行・動詞で） | 触るファイル | 手順 | 検証（完了確認） | 依存 |
|---|---|---|---|---|---|
| T1 | トークン検証関数を追加 | src/auth/token.ts | 関数 verifyRefresh を実装 | `npm test token` が green | — |
| T2 | リフレッシュ経路に配線 | src/auth/handler.ts | T1 を呼び出す | 手動 or テストで 401→200 | T1 |
| T3 | 回帰テストを追加 | test/auth.test.ts | 期限切れ→再発行ケース | `npm test` が green | T1,T2 |
```

### ルール

- **概要・未確定・タスク数行**を先頭に必ず出す（機械抽出の入口）。
- 各タスクは `T<n>` で採番。`目的 / 触るファイル / 手順 / 検証 / 依存` の各列を埋める。
- **検証列は必須**：コマンド／テスト／grep／観察で「完了したか」を言える形にする（検証の無いタスクは不可）。
- **細粒度**：1タスクは数分・少数ファイル。大きい塊は割る。過剰分割（自明な1行を3分割）もしない。
- **依存**で順序を示す（独立タスクは `—`＝並列可）。`needs:` 付き recipe へそのまま渡せる粒度にする。
- **触るファイルは具体パス**。不明なら捏造せず「未確定/要調査」に出す。
- **未確定は隠さない**：仕様の穴・前提不明は先頭の「未確定/要調査」へ。捏造でタスクを埋めない。
- 全体は簡潔に。タスクは「上から潰せば終わる」順に並べる。
