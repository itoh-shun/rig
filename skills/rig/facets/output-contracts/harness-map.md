# output-contract: harness-map

## output-contract: harness-map

ハーネス監査人（`harness-auditor`）の出力構造。プロジェクトのハーネスを 2×2 に分類し、穴を機械抽出できる固定フォーマットで出す。前置き・挨拶は禁止。

### 形式

```
## rig harness-audit: ハーネス健全性

対象: <リポジトリ/プロジェクトの1行サマリ>
総合: <健全 | 穴あり | 要改善>  ｜ 空象限 <N>/4  ｜ 穴 <M> 件（重大:<a> / 中:<b>）

### 2×2 カバレッジ
| | ガイド（先回り） | センサー（事後・検知） |
|---|---|---|
| **計算的** | <例: tsconfig strict / scaffold（有/無）> | <例: eslint・vitest・CI（有・ただしループ外 等）> |
| **推論的** | <例: CLAUDE.md・Skills（有）> | <例: PR レビュー・review-gate（有/無・独立性）> |

### 穴（重い順・各に「繋ぐ/強制する/薄くする」の手）
#### [H1] <断定形の見出し（例: テストはあるが実行ループに繋がっていない）>
- 象限: <計算的センサー 等>  ｜ 深刻度: <重大|中|低>
- 根拠: <確認した具体箇所（ファイル・設定・該当行）>
- 手: <繋ぐ/強制する/薄くする の具体策（例: PostToolUse hook で test を強制 / acceptance-gate の機械検証に追加）>

#### [H2] ...

### 最優先で繋ぐ1手
- <全部はやらない前提で、まず打つべき1手を名指し（多くは「既にある計算的センサーをループ/hook に繋ぐ」）>
```

### ルール

- **総合行**（健全/穴あり/要改善・空象限数・穴件数）を先頭付近に必ず出す（機械抽出の入口）。
- **2×2 表の4セルを必ず埋める**（無いセルは「無し」と明記＝空象限を可視化）。
- 穴は `[H<n>]` で採番し、**断定形**の見出し。`象限 / 深刻度 / 根拠 / 手` を必ず埋める。
- **手は「繋ぐ・強制する・薄くする」を優先**し、新ルールの追加は最後（善意のルール追加の逆効果を避ける）。
- **「ある」と「効いている」を区別**：存在するだけでループ/hook に繋がっていないものは穴として挙げる。
- **計算的センサー（lint/型/test）をループ外にしている穴を最優先**で拾う。
- **根拠は確認した具体箇所**。未確認は「未確認」と書く（捏造しない）。
- 末尾に「最優先で繋ぐ1手」を必ず1つ名指す。全体は簡潔に。
