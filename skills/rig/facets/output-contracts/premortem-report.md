# output-contract: premortem-report

## output-contract: premortem-report

事前検死（`pre-mortem-analyst`）の出力構造。機械抽出を前提に、失敗モードを「可能性×影響」でランクした固定フォーマットで出す。前置き・挨拶は禁止。

### 形式

```
## rig pre-mortem: 事前検死

対象: <変更/PR/計画の1行サマリ>
前提: 「数週間後、これは本番で壊れた」— なぜ壊れたかを逆算

総合リスク: <低 | 中 | 高>  ｜ 失敗モード <N> 件（高:<a> / 中:<b> / 低:<c>）

### 失敗モード（可能性×影響の高い順）

#### [R1] <断定形の見出し（例: 移行で旧データが復号できず全ユーザーがログアウト）>
- 区分: <技術的破綻 | 運用・人的 | データ・セキュリティ | 波及>  ｜ 可能性: <高|中|低>  影響: <高|中|低>
- なぜ: <そうなった原因・1〜2行>
- 兆候: <事前に見えるサイン（これが出たら危険）>
- ガードレール: <今すぐ・安く打てる具体策（テスト/フラグ/段階導入/監視/ロールバック手順）>

#### [R2] ...

### 最も安く効く 1 手
- <全部はやれない前提で、最優先で打つべき保険を名指し>
```

### ルール

- **総合リスク行**（低/中/高）と件数内訳を先頭付近に必ず出す（機械抽出の入口）。
- 各失敗モードは `[R<n>]` で採番し、**断定形**の見出し（「壊れるかも」でなく「壊れた：…」）。
- `区分 / 可能性 / 影響 / なぜ / 兆候 / ガードレール` の6項目を必ず埋める（ガードレール無しの指摘は不可＝恐怖の羅列を禁止）。
- **可能性×影響の高い順**に並べる。低×低の空想は載せない。
- 末尾に「最も安く効く 1 手」を必ず1つ名指しする。
- 捏造禁止（実際に起こりうるものだけ）。全体は簡潔に、各モード 4〜6行以内。
