# policy: cross-llm-legibility

## facet: policy / cross-llm-legibility

> **末尾注入のガードレール**。implement step の subagent prompt 末尾（recency が効く位置）に注入し、「**他社 LLM がレビューする前提**で書く」規律を効かせる。

このコードは、**Codex / GitHub Copilot / GPT など他社・他系統の LLM にレビューされる**前提で書く。基準は単純 — 「**自分（このモデル）にしか分からないコード**」を書かない。プロジェクト文脈も社内規約も知らない外部モデルが、その断片だけを読んで「正しい・慣用的・意図が明確」と判断できることを目標にする。

外部 LLM に通るコードは、結局**人間にも通る**（多様な読み手の最小公倍数＝最も素直なコード）。これは「賢いコード」より「**誰が読んでも読めるコード**」を選ぶ規律。

### 規律

1. **慣用優先**：その言語/フレームワークの**最も一般的なイディオム**で書く。トリッキーな短縮・自作の技巧・暗黙の言語仕様頼みを避ける（外部モデルは標準パターンを最も正確に認識する。珍しい書き方は「バグでは?」と疑われる）。
2. **明示性**：説明的な命名。型・戻り値・例外・nullability を明示。**暗黙の前提をコードに出す**。プロジェクト固有の暗黙知（隠れた契約・命名の慣習・魔法の定数）に依存しない（外部モデルはそれを知らない＝推測で誤読する）。
3. **自己完結**：1 関数 = 1 責務。副作用を局所化し、グローバル状態・隠れた依存を避ける。**周辺を読まずにその単位だけで正しさを判断できる**粒度にする。
4. **規約遵守**：フォーマッタ/リンタに準拠し、命名規則・ディレクトリ規約に従う（規約からの逸脱は外部モデルに「異常・意図的か不明」と映り、レビュー往復を生む）。
5. **非自明には理由**：なぜそうしたかが非自明な箇所にだけ簡潔な理由コメント（**why** を残す）。ただしコードを言い換えるだけの自明コメントは書かない（`facets/policies/` の AI-slop 排除・`de-ai-smell` と両立。何でも説明するのではなく、**読めば分かることは書かず、読んでも分からない意図だけ書く**）。
6. **テスト可能**：入出力を明確にし、外部モデルがその関数のテストを書ける形にする（テストが書けない＝契約が曖昧、のサイン）。

### 判断に迷ったら

「Codex / Copilot / GPT がこの diff をレビューしたら、**指摘ゼロで通すか?**」を自問する。通らないと感じたら、賢くするのではなく**素直にする**。
