# policy: pr-hygiene

## PR サイズ基準

| サイズ | 変更行数の目安 | 対応 |
|---|---|---|
| S | ～100行 | そのまま提出可 |
| M | ～200行 | そのまま提出可 |
| L | ～400行 | 分割を検討する |
| L超 | 400行超 | **必ず分割してから提出する** |

## 禁止事項

- **L超の PR は提出禁止。** 400行を超える変更は、関心事ごとに複数の PR へ分割すること。
- **1 PR 1 関心事。** リファクタリングと機能追加を同一 PR に混在させてはならない。バグ修正と無関係な改善も同様に別 PR にする。
- **`--no-verify` 禁止。** pre-commit / pre-push フックを `--no-verify` でスキップしてはならない。フックが失敗した場合は原因を修正してから再実行すること。
- **push 前レビュー必須。** `git push` または `gh pr create` の前に必ずローカルでコードレビューを実施すること（`/pr-pre-push-review` 等を使用）。レビューを省略したまま push することは禁止。
- **force push 禁止（保護ブランチ）。** `main` / `master` ブランチへの `--force` push は禁止。

## 推奨事項

- PR タイトルは70文字以内に収める。
- PR 本文には変更の意図（why）を含める。
- 1コミット = 1論理変更を意識し、WIP コミットは push 前に整理する。
