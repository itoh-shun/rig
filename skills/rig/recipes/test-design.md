# test-design

> **モード pack 注記**: rig engine（`SKILL.md`）を dev / magi / de-ai-smell と**共用**する QA pack の recipe。engine は書き換えず、`test-designer` persona・`qa-test-lenses` knowledge・`test-design` instruction・`test-cases` output-contract を足すだけで成立する。`/rig:qa` から起動。

## 使う場面

テストケースを**観点の抜けなく**設計したい時。AI 生成テストが**正常系に偏る**・**新機能と移行で軸がぶれる**のを仕組みで塞ぐ。例:

- 「この機能（課題#123）のテストケースを作りたい」
- 「この移行、現行どおり動くかのテストを設計して」（`--migration`）
- 「既存のテストケース表、観点の抜けをレビューして」（`--review`）

de-ai-smell（散文の AI 臭）・magi（やるか）とは目的が違う。test-design は**テスト観点とケースの設計**を担う（実行・合否は人間）。

## 7観点（取りこぼさない・各 ≥1）

| # | 観点 | 疑う対象 |
|---|---|---|
| L1 | 初見ユーザー | 素直な道・誤操作・分かりにくさ |
| L2 | 現場ベテラン | 負荷・例外フロー・大量/並行/連続操作 |
| L3 | 悪意ある操作者 | 不正入力・権限越え・改ざん（セキュリティ） |
| L4 | データ整合性監査役 | 整合性・境界値・重複/欠損・丸め |
| L5 | 移行担当者 | 旧→新の互換・引き継ぎ・後方互換 |
| L6 | 回帰デグレ番人 | 既存機能の破壊・副作用・影響範囲 |
| L7 | 仕様懐疑者 | 仕様の曖昧さ・矛盾・未定義 |

正常系（L1）を1つ押さえたら、重心を L2〜L7 へ。各観点は最低1ケース、出なければ「該当なし＋理由」。

## トラック分化

| | 新機能（既定） | 移行（`--migration`） |
|---|---|---|
| 仕様の起点 | 課題（PBI）＋ソースコード | 現行ヘルプ・現行挙動・デザイン |
| 問い | 要件を満たすか | 現行どおり動くか |
| 追加成果物 | — | 要件カバレッジ表 |

## 展開

1. **対象とトラックの確定** — 対象を1つに（1問だけ確認可）。`--migration` で移行トラック。`--plan` なら構成提示で停止。
2. **設計** — `qa-test-lenses` を注入して `test-designer` を dispatch。7観点を一巡し、根拠（Test Basis）を紐づけ、未確認は `※要確認（未実施）`、各ケースをテスト可/保留/不可に分類。
3. **構造化提示** — `test-cases`（観点カバレッジ＋ケース表＋仕様ギャップ＋移行時は要件カバレッジ表）。
4. **接続** — テスト不可/保留の仕様ギャップは仕様確定へ差し戻し。ケースの実装/自動化は `/rig:dev` へ委譲（test-design は設計まで）。

手順本体は `facets/instructions/test-design`、観点は `qa-test-lenses`、出力は `output-contracts/test-cases` に従う。

## ガード

- **正常系に偏らない・7観点を取りこぼさない**（各 ≥1 か「該当なし＋理由」）。網羅した風の水増しもしない。
- **根拠必須**（Test Basis）・**未確認は断定しない**（`※要確認（未実施）`）。捏造で「テスト可」にしない。
- **AI はテスト設計者であってテスターではない**：実行・合否判定・既存ケースの修正はしない（`--review` は指摘のみ）。
- **機密を素通ししない**（実データはダミー/伏字）・正本は消さず `DEPRECATED` 印。
