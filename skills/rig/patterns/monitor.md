# pattern: monitor

`Monitor` ツールと `until` ループを組み合わせ、バックグラウンドプロセスの完了を1回だけ検知する制御フローパターン。

## 概要

長時間実行するバックグラウンドプロセス（ビルド・テスト・デプロイなど）の完了を待つとき、`Monitor` ツールと `until` ループを使って「完了通知を1回受け取ったら次へ進む」流れを実現する。ポーリングと違い、イベント駆動で完了を検知するためリソース効率が高い。

## 使いどころ

- バックグラウンドで起動したプロセスの終了を待ってから次のステップへ進む場合。
- CI・ビルド・テストランナーなど、完了までの時間が不定なプロセスを監視する場合。
- 完了後に結果を取り出して後続処理に渡す場合。

## `tail -f` を使ってはいけない理由

`tail -f` はプロセスが終了してもファイル監視を続けるため、タスクがタイムアウトするまで制御が戻らない。長時間実行プロセスではセッションのタイムアウトを引き起こすリスクがある。代わりに `Monitor` + `until` を使う。

## オーケストレーターの実行手順

1. バックグラウンドプロセスを起動する（`Bash` の `run_in_background: true` など）。
2. `Monitor` ツールで対象プロセスのイベントストリームを購読する。
3. `until` ループで完了条件を検査する。完了条件は「プロセスの終了コード」「特定の出力行」「フラグファイルの存在」など、プロセスの特性に合わせて定義する。

```
until <完了条件チェックコマンド>; do
  sleep 2
done
```

4. 完了通知を受け取ったらループを抜け、結果を取り出して次のステップへ進む。
5. タイムアウト上限を設ける。上限を超えた場合はプロセスを終了し、user に状況を報告する。

## `until` ループの注意点

- スリープ間隔は短すぎるとポーリング過多になり、長すぎると完了検知が遅れる。通常は 2〜5 秒程度が適切。
- ループ内で副作用のあるコマンドを実行しないこと（条件チェックのみにとどめる）。
- `Monitor` ツールは完了時に通知を送るため、`until` ループとの組み合わせで効率よく完了を検知できる。
