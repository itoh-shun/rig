# pattern: autonomous-loop

`ScheduleWakeup` を使って自律的に次の実行を予約し、エージェントが長時間にわたって自律的に処理を継続するループパターン。**opt-in（`--autonomous` フラグを明示した場合のみ使用する）。**

## 概要

通常のエージェント実行はユーザーの入力を待ってターンが進む。このパターンでは `ScheduleWakeup` ツールを使って次の起動を予約することで、ユーザーの介入なしに処理を繰り返す。キャッシュの温冷によって適切な `delaySeconds` が異なる点に注意が必要。

## opt-in 制約

**このパターンは `--autonomous` フラグが明示されたセッションでのみ使用する。** 通常のフローでは使用しない。ユーザーの確認なしに自律ループを開始してはならない。

## delaySeconds の選択

| 状況 | 推奨値 | 理由 |
|---|---|---|
| キャッシュウォーム（直近で同じコンテキストを使用） | **270秒** | キャッシュヒットを最大限活用し、コスト・速度を最適化する。 |
| キャッシュミス、かつ長時間待機が許容される | **1200秒以上** | キャッシュが無い場合は十分な間隔を空けて不必要なコストを避ける。 |
| **300秒** | **禁忌** | キャッシュミスが発生するうえ待機時間が短すぎる。コスト悪化と処理の非効率が重なるため絶対に使用しない。 |

270秒はキャッシュウォーム状態での最適値。300秒を境にキャッシュが切れるため、300秒は「キャッシュミスかつ短い待機」という最悪の組み合わせになる。

## プロンプトテンプレート

自律ループで次ターンへ渡すプロンプトには必ず `<<autonomous-loop-dynamic>>` を使用する。このテンプレートは実行時に動的な状態（現在のタスク・進捗・次のアクション）を埋め込み、ループの各ターンが適切なコンテキストを持つことを保証する。

```
ScheduleWakeup({
  delaySeconds: 270,   // キャッシュウォームの場合
  prompt: "<<autonomous-loop-dynamic>>"
})
```

## オーケストレーターの実行手順

1. `--autonomous` が有効であることを確認する。有効でない場合はこのパターンを使わない。
2. 現在のキャッシュ状態（ウォーム / コールド）を判断し、適切な `delaySeconds` を選択する（270 または 1200+）。
3. 各ターンの最後に `ScheduleWakeup` を呼び出して次の起動を予約する。プロンプトは `<<autonomous-loop-dynamic>>` を使う。
4. ループを終了する条件（タスク完了・エラー・ユーザーによる中断）を必ず定義し、条件を満たした場合は `ScheduleWakeup` を呼ばずに終了する。
