# pattern: parallel-fanout

独立した複数の観点を1メッセージで N 個の subagent へ並列 dispatch し、レビュー・調査・生成を高速化する制御フローパターン。

## 概要

同一ターンで複数の `Agent` ツール呼び出しを送信することで、subagent が並列実行される。各 subagent は独立したコンテキストを持ち、互いの結果を参照せずに評価・出力する。並列化によって直列実行と比較して待ち時間を N 分の 1 に短縮できる。

## 使いどころ

- 複数の**独立した**観点（例: security / design / test）で同一の変更を評価する場合
- 独立した調査タスクを複数のエージェントに分担させる場合
- 同一の仕様に対して複数の実装案を生成し比較したい場合

**使ってはいけない場面:**

- 同一ファイルへの書き換えを複数 subagent に依頼する場合 — マージコンフリクトが発生するため必ず**直列**にする。
- subagent A の出力を subagent B が必要とする場合 — データ依存があるため直列にする。

## オーケストレーターの実行手順

1. **1つのメッセージ**に複数の `Agent` ツール呼び出しを含めて送信する（ツール呼び出しをメッセージ間で分割しない）。
2. 各 subagent に対してコンテキスト・評価軸・出力形式を明示する。出力形式は `output-contracts/review-verdict` など機械抽出可能な形式を使う。
3. すべての subagent の結果が揃うまで待機する（並列実行は自動で完了通知が来る）。
4. 全結果を受け取ったら集約処理へ進む（→ `pattern: review-gate` 参照）。

## subagent プロンプト設計の注意点

- **コンテキストを自己完結させる**: subagent は親のコンテキストを持たないため、評価対象・評価軸・出力フォーマットをプロンプトに含める。
- **役割を1つに絞る**: 1 subagent = 1観点。複数の役割を持たせると出力が曖昧になる。
- **出力フォーマットを縛る**: 機械集約を前提とするため `pattern: structured-report` が定義するフォーマットに従わせる。

## 関連ブリック

- `pattern: review-gate` — 並列結果の集約判断
- `pattern: structured-report` — 機械抽出可能な subagent 出力形式
- `output-contracts/review-verdict` — レビュー出力の共通契約
