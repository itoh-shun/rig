# pattern: structured-report

subagent の出力フォーマットをプロンプトで縛り、親オーケストレーターが結果を機械的に集約できるようにするパターン。

## 概要

自由文で出力された subagent の結果は、オーケストレーターが内容を解釈・集約する際に曖昧さを生む。このパターンはプロンプト内で出力形式を明示的に指定することで、**判定・根拠・条件・残債**を正規表現または先頭行マッチで確実に抽出できるようにする。

## 出力フォーマット

subagent に指示するフォーマット（`output-contracts/review-verdict` が定義する形式を参照）：

```
判定: <APPROVE|REJECT|APPROVE_WITH_CONDITIONS>

根拠:
1. （根拠1）
2. （根拠2）
3. （根拠3）

条件:
【マージ前必須】
- （必須対応事項。なければブロックごと省略）
【フォローアップ可】
- （任意・後続対応事項。なければブロックごと省略）

残債:
- （本タスク外で検知した懸念。なければ省略）
```

**全体 200–400字**（日本語換算）。冗長な前置き・感想・締め挨拶は禁止。

## 判定を先頭に置く理由

オーケストレーターは subagent 出力の**先頭行**を読むだけで判定を取得できる。本文を全文解析する必要がないため、集約ロジックが単純になり、出力が長い場合でも判定の見落としが起きない。

## subagent プロンプトへの組み込み方

1. プロンプトの出力指示セクションに上記フォーマットをそのまま貼り付ける。
2. 「判定を最初の行に必ず出力すること」「全体 200–400字」の制約を明示する。
3. 前置き・挨拶・感想の禁止を明記する（これを省くと subagent が冗長な導入文を出力しやすい）。

## 機械抽出の手順（オーケストレーター側）

1. 各 subagent の出力から `判定:` で始まる行を探す（先頭行のはず）。
2. その行から判定語（`APPROVE` / `REJECT` / `APPROVE_WITH_CONDITIONS`）を取り出す。
3. `条件:` ブロック以下の `【マージ前必須】` / `【フォローアップ可】` セクションを抽出して統合する（→ `pattern: review-gate` 参照）。

## 関連ブリック

- `pattern: parallel-fanout` — このフォーマットを適用する subagent 群を dispatch するパターン
- `pattern: review-gate` — 抽出した判定を集約して着手可否を決定するパターン
- `output-contracts/review-verdict` — フォーマットの正式定義（このパターンが参照する共通契約）
