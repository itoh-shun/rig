# pattern: structured-report

subagent の出力フォーマットをプロンプトで縛り、親オーケストレーターが結果を機械的に集約できるようにするパターン。

## 概要

自由文で出力された subagent の結果は、オーケストレーターが内容を解釈・集約する際に曖昧さを生む。このパターンはプロンプト内で出力形式を明示的に指定することで、**判定・根拠・条件・残債**を正規表現または行頭 `判定:` マッチ（**最後に現れる行**を採用）で確実に抽出できるようにする。

## 出力フォーマット

subagent に指示するフォーマット（`output-contracts/review-verdict` が定義する形式を参照）：

```
根拠:
1. （根拠1）
2. （根拠2）
3. （根拠3）

条件:
【マージ前必須】
- （必須対応事項。なければブロックごと省略）
【フォローアップ可】
- （任意・後続対応事項。なければブロックごと省略）

残債:
- （本タスク外で検知した懸念。なければ省略）

判定: <APPROVE|REJECT|APPROVE_WITH_CONDITIONS>
確信度: <高|中|低>
```

**全体 200–400字**（日本語換算）。冗長な前置き・感想・締め挨拶は禁止。

## 判定を末尾に置く理由（evidence-first）

判定を先頭に出させると、subagent は理由付けを書く前に結論へコミットし、以降の根拠が結論の後付け正当化に流れやすい（LLM-as-judge の既知バイアス）。根拠→判定の順に縛ることで、証拠を検討してから結論を出す推論順を強制する。機械抽出は行頭 `判定:` マッチのままなので、集約ロジックの単純さは失わない——本文中に判定行の引用が混ざっても、**最後に現れる** `判定:` 行を採用すれば見落とし・誤抽出は起きない。

## subagent プロンプトへの組み込み方

1. プロンプトの出力指示セクションに上記フォーマットをそのまま貼り付ける。
2. 「根拠を先に書き、判定と確信度を最後の行に必ず出力すること」「全体 200–400字」の制約を明示する。
3. 前置き・挨拶・感想の禁止を明記する（これを省くと subagent が冗長な導入文を出力しやすい）。

## 機械抽出の手順（オーケストレーター側）

1. 各 subagent の出力から `判定:` で始まる行を探し、**最後に現れる行**を採用する（末尾にあるはず。根拠中に引用された判定行に釣られない）。
2. その行から判定語（`APPROVE` / `REJECT` / `APPROVE_WITH_CONDITIONS`）を取り出す。
3. `条件:` ブロック以下の `【マージ前必須】` / `【フォローアップ可】` セクションを抽出して統合する（→ `pattern: review-gate` 参照）。
