# rig — LEGO 式ハーネス・コンポーザ

## これは何か

rig は Claude Code のための「レゴ式」開発フロー・オーケストレータ。固定のワークフローを押しつけるのではなく、ブリック（facet・pattern・recipe・persona）を**起動時に組み合わせて**タスク専用のエージェント・ハーネスを毎回 engineering する。

たとえば「軽い変更だから review だけ並列で回したい」「本番影響変更だから design → implement → review → PR まで通したい」「Issue 番号を渡して全工程を自律で」など、要求ごとに異なるハーネスが立ち上がる。

## 4 段プロセス: PARSE → RESOLVE → COMPOSE → RUN

rig はあらゆる起動を4段で処理する。

1. **PARSE**: 起動文字列（フラグ・自由記述）を解釈する。`--review`・`--tdd`・`--autonomous`・`--issue 42`・`--recipe release-flow` などを切り分ける。
2. **RESOLVE**: manifest（`.claude/rig.md`）・recipe・flag・変更規模（size-aware）を順に適用してハーネスを確定する。後の段が前の段を override する。recipe は project（最高）→ user → shipped の順に解決される。
3. **COMPOSE**: 確定した step ごとに persona（System）・knowledge（User 先頭）・instruction（User 中部）・output-contract（User 構造）・policy（User 末尾）を facet 配置順厳守で組み立て、subagent prompt を生成する。
4. **RUN**: Claude Code の native primitive（Agent ツール、Task、skill）で実行する。並列可能な観点は parallel-fanout で 1 メッセージ多 dispatch、集約は review-gate。

## context-minimal — ハードルール

rig の核心は **context-minimal**。親（オーケストレーター）は dispatch と structured-report の集約とゲート判断だけを行い、実作業（実装・レビュー・調査・デバッグ）は必ず subagent に dispatch する。長い tool 出力やコード本文を親 context に引き込まない。

これにより、親の文脈は常に小さく保たれ、大きな変更でも context overflow せずに最後まで走り抜ける。

## determinism-by-gate — 品質の保証

LLM の実行は非決定的だが、`acceptance-gate` パターンで挟むことで毎回同じ品質に収束する。受け入れ基準（build 成功・lint 0 件・review に REJECT が無い等）を満たすまで最大 K 回まで自動収束。経路は変動しても**毎回同じ品質**へ着地する。

## 並列レビュー — 1 メッセージで 3 観点を同時に

`/rig:dev --review` で security / design / test の 3 観点が**並列に**走る。それぞれが独立した reviewer subagent として起動し、structured verdict（review-verdict 契約）を返す。親はその判定行だけを読み、REJECT が無ければ通す。

`--adversarial` を付けると lazy-senior・cognitive-economist の敵対的レビュー（AI の癖排除・人間可読性・不要コメント除去）が追加される。`--persona <name>` で custom reviewer も差し込める。

## before / after — Claude Code 単体 vs rig

**before (Claude Code 単体)**: ユーザーが「この機能を実装して」と頼むと、Claude が一人で全部やる。実装も、レビューも、PR も。途中で context が膨らみ、後半は判断が雑になる。レビュー観点も「気になったところ」止まり。

**after (rig)**: 同じ依頼が `/rig:dev` で受けられる。親は「これは S サイズなので review だけ並列で」と判断し、3 reviewer subagent に dispatch する。実装そのものも implementer subagent に投げる。親の context は常に空。最後まで判断が鈍らない。レビューは security / design / test の 3 観点が必ず走る。

## brick の在庫

- **persona** （reviewer 人格・観点）: security-reviewer, design-reviewer, test-reviewer, lazy-senior, cognitive-economist など
- **pattern** （制御フロー）: parallel-fanout, review-gate, acceptance-gate, serial, autonomous-loop など
- **recipe** （step の束）: review-only, release-flow, design-first, hotfix, adversarial-review, goal-loop など
- **policy** （末尾ガードレール）: pr-hygiene, branch-strategy, risk-based-testing など

これらをタスクに応じて起動時に組み合わせる。固定 DSL は無い。
