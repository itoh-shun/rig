# spd-stakeholder-review — SPDステークホルダー・レビュー

SPD = **Supply Processing and Distribution**。病院の診療材料・医薬品・消耗品の購買・在庫（定数管理）・供給・消費データ管理を一元化する業務。業界団体は一般社団法人 日本SPD協議会（https://www.spdjapan.org/ ）。ドメイン知識は同梱の `references/knowledge/` が正本。

## 2つのモード

| モード | いつ使うか |
|---|---|
| **6視点レビュー**（既定） | 提案書・仕様・業務フロー・契約骨子などの成果物を、業界に登場する6つの立場から同時にレビューしたい時 |
| **単一ペルソナ相談**（`--as <persona>` または「◯◯の立場で」と依頼された時） | 特定の立場（例: 看護現場）の目線で質問・壁打ちしたい時 |

## 登場ペルソナ（6ステークホルダー）

| persona | 立場 | 主な関心 |
|---|---|---|
| `hospital-executive` | 病院経営層・事務長 | 費用対効果・経営指標・ガバナンス |
| `materials-manager` | 用度課・材料部（購買） | 物品マスタ・契約と責任分界・検収・価格の見える化 |
| `ward-nurse` | 看護現場（病棟師長） | 欠品ゼロ・補充の確実さ・現場負担・安全 |
| `spd-operator` | SPD現場責任者（センター長） | 運用KPI・例外処理・人員の現実性・定数見直し |
| `spd-vendor-manager` | SPD事業者の経営者 | 採算・契約範囲の明確さ・価値の言語化・持続性 |
| `distributor` | 医療材料卸・ディーラー | 供給実現性・預託在庫の負担・GS1標準・制度整合 |

定義本体は `references/personas/<persona>.md`。

## 手順（6視点レビュー）

1. **対象の受理** — 本文またはファイルパスで評価対象を受け取る。形式は問わない。欠落があっても記入を強制せず、推測で補完しない（欠落は各ペルソナが「情報不足」として指摘する）。
2. **知識の読み込み** — `references/knowledge/` の `spd-basics.md` / `spd-industry.md` / `spd-glossary.md` を文脈として使う。`domain-template.md` に自院/自社固有の記入があれば併用する。
3. **6視点の並列評価** — 6つのサブエージェント（並列実行できない環境では順次でよい）に、それぞれ `references/personas/` の該当ペルソナ定義＋知識＋評価対象を渡し、**自分の1視点だけ**を評価させる。出力形式は各ペルソナ定義にインラインで書かれた形式（観点/根拠/良い点/懸念・要改善点/情報不足/判定◎○△×）に従わせる。
4. **収束** — 次の基準をすべて満たすまで不足分を再評価させる: ①6視点すべてが判定済み（◎○△×）、②各懸念・要改善点が「誰が次に何をすべきか」が分かる粒度、③情報不足と事実誤認（制度・償還・薬機法まわり）が明示されている。
5. **集約提示** — `references/verdict-format.md` の集約フォーマットに従い、総合判定（GO / 条件付きGO / 要再検討）＋視点別テーブル＋優先アクション＋情報不足を提示する。情報不足が過半で評価不能なら「記述不足のため評価保留」とし、不足項目の追記を促す。

## 手順（単一ペルソナ相談）

指定されたペルソナ1体の定義と知識を読み込み、**その立場から**会話体で質問・相談に答える。集約フォーマットは課さない。ただしペルソナの規律——評価・助言のみで実装はしない、推測で埋めず情報不足を明示する——は維持する。

## 自院 / 自社固有の文脈

`references/knowledge/domain-template.md` に施設プロフィール・運用実態・KPI・課題を記入すると、各ペルソナが固有文脈で評価する。未記入なら汎用の業界知識のみで動く。
