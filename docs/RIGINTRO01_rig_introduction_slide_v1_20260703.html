rig — タスクごとに
「足場」を組み替える開発ハーネス・コンポーザ

固定のワークフローを1本持つのではなく、部品（ブリック）をその場で組み合わせて、タスク専用のエージェント・ハーネスを毎回組み上げる Claude Code プラグイン。
本スライドは「rig の用語がわかりにくい」を解消することを目的に、用語辞典（一文定義＋たとえ）を中核に据えて rig を紹介する。

What is rig

02rig とは何か——1行と、その背景

「素の AI エージェント」の弱点から出発する
DEFrig ＝ ブリック（部品）を起動時に動的に組み合わせ、タスクに最適化されたエージェント・ハーネスを engineering する、レゴ式ハーネス・コンポーザ。

課題：AI に任せると、毎回ばらつく

  • 同じ依頼でも、日によって進め方・品質・確認の深さが変わる
  • レビュー観点が抜ける・自分の仕事を自分で「OK」と甘く判定する
  • タスクの種類（実装／レビュー／ホットフィックス／設計）ごとに最適な進め方は違うのに、道具は1本の固定フローになりがち

rig の答え：組み替える。そして出口で締める

  • 組み替える——タスクを読んで、必要な部品（観点・手順・型・ルール）だけをその場で合成する。3-Stage フルフロー（intake→design→implement→verify→review→pr→merge）も「数あるレシピの1つ」にすぎない
  • 出口で締める——実行の経路は揺れてよい。代わりに受け入れゲート（acceptance-gate）が毎回同じ基準で出口を締め、同じ品質へ収束させる
  • 重い DSL・外部依存なし。Claude Code ネイティブ（command＋skill＋agents）

名前の由来は「rig up ＝（船や舞台の）艤装・仕掛けを組むこと」。モデル（頭脳）は変えられないが、その周りに組む足場は設計できる——という Agent = Model + Harness の考え方を、そのまま道具にしたもの。

Core idea

03中心思想：determinism-by-gate

「非決定的な実行」を「決定的なゲート」で挟む——rig の品質保証の核
図：determinism-by-gate。モデルの自由（裁量）は残し、出口の基準だけを決定的に固定する。
縛らない、でも通さない。モデルをルールで縛るほど賢さは死ぬ。rig は実行を縛らず、「何をもって合格か」だけを機械が判定できる形で固定する。差し戻し回数に上限（K 回）を置き、超えたら人へエスカレーション——無限ループも防ぐ。
採点者 ≠ 生成者。作った本人に自分の仕事を採点させない。レビューは独立した別エージェント（read-only・専用コンテキスト）が行い、さらに反証者（finding-verifier）が指摘の側もチェックする（S9）。

04用語の見取り図——3グループに分ければ迷わない

rig の用語は多いが、役割は「部品」「組み立て」「実行と品質」の3つしかない

Glossary 1/3

05用語辞典① 部品——「ブリック」の中身

ブリック＝rig を構成するレゴブロックの総称。その主な種類が以下
用語ひとことで（正確な定義）厨房でいうと
ブリック（brick）rig を構成する部品の総称。足したり外したりして足場を組む。エンジン本体は変えず、ブリックを追加するだけで新しいフローが乗る食材・道具・レシピカードなど、厨房にある「組み合わせ可能なもの」全部
facet（ファセット）作業担当のエージェントに渡すプロンプトへ合成される「面」。下の persona／instruction／policy／knowledge／output-contract は全部 facet の種類調理担当に渡す指示書の「記入欄」——役割欄・手順欄・禁止事項欄・参考資料欄・盛り付け見本欄
persona（ペルソナ）レビュアーなどの人格・観点の定義。「セキュリティ観点」「テスト観点」など、何をどう見るかを1枚に固めたもの。/rig:persona で自作もできる観点の違う味見役——塩加減担当・アレルギー担当・盛り付け担当。増やせば見落としが減る
agent（エージェント）persona の独立実行版。専用のコンテキストとツール制限（read-only）つきで起動される native のレビュアー。あれば agent 優先、なければ persona で代用外部から呼ぶ覆面の試食審査員。厨房の事情（実装の経緯）を知らずに皿だけを見る
instruction手順書。ただし厚い手順を抱え込まず、既存の skill／command／agent へ「薄く委譲」する routing が原則工程カード。「ソースは3番レシピ参照」のように、詳細は正本へ飛ばす
policy（ポリシー）プロンプト末尾に注入されるガードレール（例：PR 衛生・push 前レビュー・CI コスト配慮）壁に貼った「揚げ物は二度確認」の類の掟。どの料理にも最後に効く
knowledge／wiki知識ページ。1概念＝1正準ページの LLM-wiki（出典・確認日必須）として持ち、persona には事実を埋め込まず inject: [[ページ名]] で参照させる＝知識を暗黙知にしない厨房の共有レシピブック。各自の頭の中でなく、棚の正本を全員が参照する
output-contract出力の契約。レビュー結果などを機械が確実に読み取れる固定フォーマットで返させる定義（例：review-verdict）検品票の様式。自由作文でなく、合否・理由・必須条件を決まった欄に書かせる

Glossary 2/3

06用語辞典② 組み立て——部品を束ねて再利用する

1回組んだ足場は、名前をつけて何度でも使い回せる
用語ひとことで（正確な定義）厨房でいうと
step（ステップ）フローの工程単位。intake（受付）／design（設計）／implement（実装）／verify（検証）／review（レビュー）／pr／merge など下ごしらえ→調理→味見→盛り付け→提供、の各工程
pattern（パターン）step をどう回すかの制御テンプレ。並列に撒く（parallel-fanout）・直列（serial）・検品を挟む（review-gate／acceptance-gate)・自律ループ（autonomous-loop）など段取りの型。「3品同時進行」「1品ずつ」「必ず検品を挟む」
recipe（レシピ）step＋pattern＋facet を固定したテンプレのワークフロー。release-flow・review-only・hotfix など。extends で親レシピを継承、checks:（lint／test 等の機械検証）や needs:（工程間の依存）も書ける。--save-recipe で今回の合成を保存できる文字どおりレシピ。「コース仕立て」「まかない速攻版」を名前で呼び出す
pack（パック）エンジンを一切変えずにブリック一式を上乗せするドメイン拡張。sales（営業）／goal（ゴール駆動）／magi（3賢者合議）／design（UI/UX）など。「engine 不変・ブリック追加のみ」が原則メニュージャンルの追加。和食パックを足しても厨房の設備は同じ
manifest（マニフェスト）プロジェクトごとの方針・既定値を書くファイル（.claude/rig.md）。常時投入するレビュアー（default_personas）・常時カッチリ実行（default_orchestrate）などを固定する。/rig:init が雛形を生成店の方針ノート。「うちは必ずアレルギー確認」「検品は毎回機械式」
tier（ティア）／overlayブリックの置き場の3層：shipped（rig 同梱）→ user（自分の全プロジェクト共通）→ project（このリポジトリ専用）。近い層が勝つ解決順（project→user→shipped）で、同名 recipe はプロジェクト側が上書き（overlay）できるチェーン共通レシピ＜自分の定番＜この店のアレンジ。近い方が優先
覚え方：facet／pattern／step が「素材と段取り」、recipe がそれを束ねた「一皿の作り方」、pack が「ジャンル一式」、manifest／tier が「店ごとの流儀と置き場」。

07用語辞典③ 実行と品質——回し方・締め方の言葉

ここが rig の独自色。「非決定を決定で挟む」ための語彙
用語ひとことで（正確な定義）厨房でいうと
PARSE → RESOLVE →
COMPOSE → RUN		エンジンの4段。読む→選ぶ→組む→回す（詳細は次スライド）注文を聞く→レシピと材料を決める→調理台を整える→調理する
acceptance-gate受け入れゲート。step の出口に置く決定的な合格判定。REJECT なら差し戻し、K 回超えたら人へエスカレーション（determinism-by-gate の実体）お客に出せる皿の検品基準。基準未満は作り直し、揉めたら店長を呼ぶ
fan-out（ファンアウト）レビューを複数の観点へ並列に撒くこと。既定は 3-way（security／design／test）、選択投入や --persona で増やせる（全11枠＋）1皿を味見役3人に同時に回す。観点が違うから見落としが重ならない
finding-verifier
（反証者）		レビュアーの指摘の側を検証する独立エージェント。証拠つきで反証された指摘（REFUTED）はゲートに通さない＝誤検出（false-positive）の最終制御「その指摘、本当？」を確かめる二次検品。味見役の言いがかりを弾く
orchestrate
（計算的オーケストレーション）		工程の遷移・ゲート判定・リトライ・停止・状態保持を、散文（LLM の判断）でなくコード（決定論ランナー orchestrate.py）に強制させる実行モード。--orchestrate 明示のほか、recipe に checks:／needs: があれば自動 ON人の勘でなく、キッチンタイマーとチェックリスト装置が工程を進める
context-minimalRUN の絶対条件。各担当には必要最小の文脈だけを渡す（全部盛りは精度もコストも悪化させる）担当者には担当の分の指示だけ渡す。厨房全体の事情は背負わせない
run-continuity質疑や脱線・コンテキスト自動圧縮を挟んでも rig の駆動が切れないための仕組み（毎ターンの状態ヘッダ再掲・再アンカー・PreCompact フック）中断後も「今どの注文のどの工程か」を必ず声に出してから再開する
capture（捕捉）RUN 後に得た学び（AI の癖・落とし穴・決定）を承認つきで知識層に書き溜める学習サイクル。サイレント書き込みは禁止まかないノート。今日の失敗と対策を残し、次の営業に効かせる
telemetry／drill全実行の記録（.rig/runs.jsonl）と、レビュアー検出率の実測（既知のバグの種を注入して検出／見逃し／誤検出を採点＝/rig:drill）営業日報と、覆面調査（わざと異物を混ぜて検品が見つけるか試す）

Engine

08エンジンの4段：PARSE → RESOLVE → COMPOSE → RUN

どのコマンドで呼んでも、内側ではこの4段が動いている

「都度合成」だから起きること

  • 同じ /rig:dev でも、レビューだけ（--only review）・TDD（--tdd）・敵対レビュー追加（--adversarial）と、組まれる足場が毎回変わる
  • 気に入った構成は --save-recipe で名前をつけて固定できる

実行前に「見える」

  • --plan＝組んだハーネスを人間可読で提示して停止（実行しない）
  • --list＝部品と recipe の在庫一覧、--validate＝部品の整合点検（doctor）

Quality

09品質の仕組み：レビューを「多観点×独立×反証」にする

1人の頭の中の「たぶん大丈夫」を、構造で置き換える

STEP 1多観点に撒く（fan-out）

  • 既定 3-way：security／design／test
  • 選択投入：performance・observability・api-compat・migration・docs
  • 敵対枠：lazy-senior・cognitive-economist（AI の癖・過剰実装を突く）
  • 自作 persona も --persona で投入可

STEP 2独立させる

  • レビュアーは read-only の別エージェント——実装の経緯を知らず、成果物だけを見る
  • orchestrate の全自動モードでは、検証を別プロバイダ（別のモデル）に振ることもでき、構造的に「採点者≠生成者」になる

STEP 3指摘の側も検証する

  • --verify-findings：REJECT の根拠を1件ずつ反証者（finding-verifier）に渡す
  • 証拠つきで反証された指摘（REFUTED）はゲートに通さない
  • 「疑わしきは所見の利」＝UPHELD／UNRESOLVED は通す
結果の読み取りも機械式：各レビュアーは自由作文でなく output-contract（review-verdict）の様式で返すため、ゲートは合否・必須条件を確実に拾える。品質を「人の注意」でなく「構造」で守る——という手元資料（CLAUDE-00 §09）の指針を、レビューに実装した形。
やり過ぎ防止も内蔵：レビュアーを増やすほど誤検出と費用も増える。--budget low/mid で fan-out を間引き、telemetry（runs --personas）で「5票以上 REJECT ゼロ＝ゴム印疑い」のような剪定ヒントが出る。

Two ways to run

10回し方は2つ：散文エンジンと、決定論ランナー

「サッと対話で」か「毎回同じ水準でカッチリ」か——舵を誰が握るかの違い

既定：散文エンジン（舵＝LLM）

  • SKILL.md の手順を読んだ Claude 自身が工程を進める
  • 軽い・即時・対話的。一発のレビューや小さな実装に向く
  • ゲートやリトライも効くが、進行判断はモデルに委ねる

--orchestrate：計算的オーケストレーション（舵＝コード）

  • 遷移・ゲート判定・リトライ・停止・状態保持を scripts/orchestrate.py（決定論ランナー）が強制する
  • 全自動モードでは各 step を別プロセスのエージェントで実行——並列検証・工程 DAG・マルチプロバイダ（claude／codex／ollama 等）・worktree 隔離（run --isolate）
  • 状態は run-state.json に永続＝止めても続きから
こういう時orchestrate？理由
普通に /rig:dev を一発・対話でOFF（散文）軽い・即時が正義
recipe に checks:（lint／test）や needs:（依存）がある自動 ON機械検証・DAG は決定論で回す意図
manifest に default_orchestrate: true常時 ONこのプロジェクトは毎回カッチリ
今回だけ軽く戻したい--no-orchestraterun 単位で打ち消せる

自動で切り替わった時は理由と戻し方を1行で通知（例：🧭 計算的オーケストレーションで回します（理由: recipe に needs 宣言））。黙って挙動が変わらないのも設計方針。

Goal & Loop

11ゴール駆動とループ：任せて、席を外せるように

「達成するまで回る」goal と「またの機会に回る」loop——役割が対になっている

/rig:goal — 達成するまで回る（内向きのループ）

  • 高レベルな目標を渡すと、まず受け入れ基準（＝測れる完了条件）に変換する
  • 「現状把握 → 次の一手 → 既存フローへ委譲 → 基準と照合」を達成まで周回
  • policy independent-verification（採点者≠生成者）を同梱——判定役の独立はここでも原則
  • 例：/rig:goal "ログイン不具合を回帰込みで直して review 通過まで"

/rig:loop — いつまた回るかを担う（外向きのループ）

  • watch／poll／repeat＝監視・定期の繰り返し。goal の対極
  • 停止条件（--until／--times）と安全上限が必須・各周回で報告
  • --every で /rig:goal を定期キック＝外側のスケジューラとして重ねられる
手元資料（GENAI-E01）との対応：これはループエンジニアリングの原則——ゴールは測れる形で・判定役は独立・停止条件は必須・状態は置き場に記録——をコマンドにしたもの。ループの土台になる環境（ハーネス）は S5〜S10 の部品が既に用意している、という積み上げ構造。
途中で切れない工夫（run-continuity）：長い run では質疑・脱線・コンテキスト自動圧縮が必ず挟まる。rig は毎ターン状態ヘッダ（▸ rig | recipe … | step … | gate …）を再掲し、圧縮時は PreCompact フックが run 状態の保全を注入——「いつの間にか素の Claude に戻っていた」を防ぐ。

Measure & Grow

12測って、育てて、還元する

足場は組んで終わりではない——数字で測り、学びを貯め、外とやり取りする

測る

  • telemetry：全 run が .rig/runs.jsonl に自動記録。DONE 率・平均リトライ・レビュアー別の票を集計
  • /rig:drill：既知のバグの種を使い捨て worktree に注入 → レビュアー別の検出／見逃し／誤検出スコアボード＝判定役の品質を意見でなく数字に
  • /rig:party 🎮：在庫と実測を RPG のパーティ画面風に描画——見た目は遊び、全行が実データの健康診断

育てる

  • capture：run で得た学び（AI の癖・落とし穴・決定）を承認つきで知識層へ蓄積
  • /rig:persona：説明文からレビュアー観点を自作
  • /rig:knowledge：ドメイン知識を wiki ページ化（web 調査 --research・リポジトリ解析 --auto）
  • /rig:forge：ブリック／パック自体を自作する自己拡張

還元する（吸収と還元の輪）

  • /rig:import：ネット上の外部 skill を検疫→判断→検証つきで取り込む（出所と SHA-256 を lock 記録）
  • /rig:export：rig で育てた部品を、rig を知らない人もそのまま使える独立 skill に書き出す
  • /rig:catalog：全層の部品を横断した地図——「誰がどこで何をしているか」を取り戻す
ここが一番の差別化：多くの道具は「組む」まで。rig は検品係（レビュアー）そのものを覆面調査（drill）で採点し、働きの悪い味見役をデータで入れ替えるところまで踏み込む。ハーネス→ループの次に来る「判定基準を体系化する」段階（eval engineering）の実装例。

Commands

13入口マップ：主要コマンドを役割で分ける

コマンドは多いが、役割グループは5つ。まずは太字から
グループコマンド使いどころ
つくる（本流）/rig:dev（開発フロー）・/rig:tasks（タスク分割）・/rig:brainstorm（設計の壁打ち）・/rig:queue（積んで一括実行）実装・レビュー・PR 完了など日常の開発。迷ったら /rig:dev
まわす（自律）/rig:goal（達成まで）・/rig:loop（定期・監視）・/rig:orchestrate（決定論ランナー）任せて席を外したい・毎回同じ水準で回したい

Get started

14はじめかた ＆ まとめ

3行で入れて、3語で覚えて帰る

インストール（marketplace 経由）

/plugin marketplace add itoh-shun/rig
/plugin install rig@itoshun-local-plugins

最初の一歩

/rig:init                 # プロジェクトを初期化
/rig:dev --plan "変更をレビュー"   # まず組まれる足場を見る
/rig:dev --only review    # レビューだけ回してみる

--plan で「実行せず中身を見る」から入るのが安全な順序。

持ち帰る3語

  • ブリック——rig の部品。persona（観点）・recipe（工程の束）・pattern（段取りの型）… すべて足し外しできるレゴブロック
  • determinism-by-gate——実行は縛らず、出口の合格基準だけを毎回同じに固定する。rig の品質保証の核
  • 採点者≠生成者——作った本人に採点させない。独立レビュー＋反証者＋覆面調査（drill）で、判定役まで疑って測る

3つのエンジニアリングとの対応

  • プロンプト＝部品化して版管理（persona・instruction・wiki）
  • ハーネス＝タスクごとに合成（COMPOSE・gate・policy）
  • ループ＝goal／loop／orchestrate で「達成まで・定期に」回す
