3つのエンジニアリング整理
——プロンプト／ハーネス／ループ

生成AI活用の実務で並んで語られる3つの「エンジニアリング」——プロンプトエンジニアリング・ハーネスエンジニアリング・ループエンジニアリング——を、それぞれ「何を最適化する営みか」「誰がいつ言い出したか」「Claude ではどの部品にあたるか」の3点で整理する資料である。 3語は流行語の言い換えではなく、最適化の対象が「一回の応答」→「毎回の実行環境」→「繰り返しそのもの」へと一段ずつ外側に広がってきた系譜として読むと見通しがよい。

記述は Anthropic の公式ドキュメント（プロンプトエンジニアリングガイド・Claude Agent SDK 解説・コンテキストエンジニアリング解説）と、各語の考案者・提唱者の一次情報（Mitchell Hashimoto、OpenAI の実践報告、Martin Fowler 誌の体系化記事、2026年6月のループエンジニアリング提唱群）に依拠し、手元の資料群（GENAI-01／GENAI-G01 用語集／CLAUDE-00／STEPUP-CLAUDE-01）の枠組みへ接続する。出典は §06 に集約。

公式／通称の区別（重要）：3語のうち「プロンプトエンジニアリング」だけが Anthropic の公式用語（公式ドキュメントに定義と体系がある）。「ハーネス」は公式用語だが「ハーネスエンジニアリング」は通称（2026年2月に実務家発で定着）、「ループエンジニアリング」も通称（2026年6月に定着）。通称側は定義が発話者により揺れるため、本資料では提唱者の一次情報の定義を採る。

01 なぜ3つ並ぶのか最適化の対象が一段ずつ外側へ広がった

3語は対立概念ではなく、生成AIの使い方が「対話 → 委任 → 自律」へ深まるにつれ（STEPUP-CLAUDE-01 の階段そのもの）、設計対象が入力 → 環境 → 反復へと外側に拡張されてきた時系列にある。間に挟まる「コンテキストエンジニアリング」は、Anthropic 自身が「プロンプトエンジニアリングの自然な発展形」と位置づける中間段である。

図1：3つのエンジニアリングの系譜。新しい語が古い語を置き換えるのではなく、外側の層が増えるたびに内側の技芸はその部品として埋め込まれる（§05）。

02 プロンプトエンジニアリング公式用語（Anthropic）

望む出力を一貫して得られるよう、モデルへの入力を構造化する営み。指示の言い回し、情報を置く順序、依頼の形式まで含む——Anthropic 公式ドキュメントの定義。

なぜこの営みが要るのか

同じモデルでも、依頼の書き方ひとつで出力の質・形式・安定性が大きく変わる。「うまく書けば良い答えが返る呪文」ではなく、役割・文脈・制約・例を組み合わせて設計する対象であり（GENAI-G01 用語集「プロンプト」）、その設計の巧拙が最初に効く変数だから。生成AI活用の基礎スキルとして最初に押さえる位置にある。

公式ガイドが置く「前提」——書き始める前に決めること

Anthropic のプロンプトエンジニアリングガイドは、テクニックの前に次の3点を前提として要求する。ここが本資料全体の伏線になる——「成功基準を先に決め、実証的に測る」という姿勢は、後のループエンジニアリングの完了条件設計にそのままつながる。

  1. 成功基準の明確な定義——何をもって「良い出力」とするか。
  2. 基準に対する実証的なテスト手段——目視の印象でなく、評価（eval）で測れること。
  3. 改善したい初稿プロンプト——ゼロからでなく、たたき台を回して磨く。

また「いつプロンプトエンジニアリングをするか」も公式に線が引かれている：プロンプトで制御できる成功基準に対して行う。レイテンシやコストは、プロンプトの工夫よりモデル選択（軽量モデルへの切替）の方が効くことがある——「何でもプロンプトで解こうとしない」が公式の姿勢である。

代表的な技法（公式ガイドの体系）

技法一言でいうと
明確・直接的に書くタスク・読み手・完了像を明示する。曖昧な依頼に良い出力は返らない
例を与える（multishot）期待する入出力の例を数個見せる。形式のばらつきに最も効く
考えさせる（chain of thought）結論の前に段階的に考えさせ、複雑な課題の精度を上げる
XMLタグで構造化指示・資料・例を <instructions> 等のタグで区切り、混線を防ぐ
ロールを与える（システムプロンプト）役割・前提・口調を会話の土台として固定する（ユーザー発言より上位で効く層）
応答の書き出しを与える（prefill）出力の先頭を指定して形式・方向を固定する
チェーン化（prompt chaining）大きな課題を複数の小プロンプトに分割し、順に渡す

発展形：コンテキストエンジニアリング

Anthropic は2025年9月の技術解説で、コンテキストエンジニアリングを「プロンプトエンジニアリングの自然な発展形」と位置づけた。プロンプトエンジニアリングが「指示をどう書くか」なら、コンテキストエンジニアリングは「推論のたびにどのトークン集合を文脈に保つか」——書き方から、取捨選択と維持管理へ。背景には、注意予算・コンテキスト腐敗（用語集C群）という「足すほど効く、は成り立たない」性質がある。エージェント時代には後者の比重が増す。

〈日常のたとえ〉

誰かに仕事を頼むときの依頼メモの書き方。「いつまでに」「どんな形式で」「この例のように」と書くほど期待通りに動いてもらえる。コンテキストエンジニアリングは、そのメモに添える資料をどう厳選するか——山ほど渡せば良いわけではなく、肝心の1枚が埋もれない分量に絞る側の技術。

使いどころ

出力がズレたら、役割・文脈・制約・例のどれが足りないかを点検する。単発の作業なら、この営みだけで十分に完結する（STEPUP ステップ1〜2 の主戦場）。ただし「毎回同じ工夫を書き直している」と気づいたら、それは次のハーネスへ移す合図。

03 ハーネスエンジニアリング通称（2026-02 定着）

モデルそのものではなく、モデルを囲む環境——前提知識・規約・権限・道具・検証の仕組み一式（＝ハーネス）——を設計し、エージェントが一度した失敗を構造的に繰り返せないように育てていく営み。

誰が言い出したか（考案・定着の経緯）

  • 2026-02-05 Mitchell Hashimoto（HashiCorp 共同創業者・Terraform 作者）がブログ「My AI Adoption Journey」で自身の規律を言語化：エージェントがミスをするたび、その特定のミスが二度と起きないよう恒久的な修正を環境側に組み込む。数日後、Agent = Model + Harness（エージェント＝モデル＋ハーネス）という定式で要点を蒸留し、実務家に一気に広まった。
  • 2026-02-11 OpenAI の Ryan Lopopolo が実践報告を公開：手書きコード0行のまま5ヶ月で本番アプリを構築（最終約100万行・約1,500件の自動PR）。「最も困難な課題は環境・フィードバックループ・制御系の設計に集中していた」と述べ、この語に実績の裏づけを与えた。
  • Martin Fowler 誌（Thoughtworks の Birgitta Böckeler）が体系化記事「Harness engineering for coding agent users」を公開し、ハーネス部品を guides（ガイド）と sensors（センサー）に分類する語彙を定着させた。

なお「ハーネス」という語自体は Anthropic も公式に使う（用語集A群：モデル核を囲み、ツール・ルール・実行手順・記憶を与えて安全に働かせる仕組み一式）。通称なのは「〜エンジニアリング」と呼んで一つの技芸に格上げした部分である。

体系：guides（事前に導く）と sensors（事後に検知する）

区分働きClaude での代表部品
guides

Claude Code はハーネスの実物である

CLAUDE-00 §04 の通り、第三者解析（論文 arXiv:2604.14228、非公式）は Claude Code のソースの約98.4%が「考える部分」ではなくハーネスだとする。設計思想は「モデルを縛る」の逆で、モデルに大きな裁量を与え、周囲の環境を徹底して作り込む。実務上とくに重要なのが次の分離：

ガイダンスと強制の分離：CLAUDE.md は「お願い」として渡り、確実な遵守は保証されない。確実に守らせたいルールは、決定論的に効く側——permissions（Deny → Ask → Allow）・hooks・サンドボックス——で縛る。「お願い（guides の一部）」と「強制（歯止め）」を混同しないことが、ハーネス設計の第一原則になる。

〈日常のたとえ〉

新入社員が独り立ちできるように整える支援体制一式（マニュアル・チェックリスト・権限・相談体制）。本人の地頭（モデル）だけでなく体制の作り込みで成果が決まる。ハーネスエンジニアリングは、その体制を失敗のたびに恒久改善していく本部の仕事——個々の担当者を鍛えるのではなく、誰が担当してもうまく回る仕組みを育てる側に立つ。

使いどころ

「同じ注意を毎回プロンプトに書いている」「エージェントが同じ失敗を繰り返す」が着手の合図。プロンプトの言い回しを磨くより、環境（前提・権限・自動チェック）を1つ直す方が、以後のすべてのセッションに効く。STEPUP ステップ3.5 の主戦場で、コード的な構築（ファイル・権限・hooks）に入るため非エンジニアには壁になりやすい——その場合は Claude 単体＋外部化の「仕組み化」（経路B）で近づく道もある。

04 ループエンジニアリング通称（2026-06 定着）

目標と完了条件をあらかじめ定義し、エージェントの「実行→検証」の繰り返し（ループ）そのもの——どこで回し、何を確かめ、どこで止めるか——を設計する営み。人は毎回のやり取りの外に出て、ループを設計・監督する側に回る。

公式の土台：agentic loop

Anthropic は Claude Agent SDK の公式解説で、エージェントの基本動作を 「文脈収集（gather context）→ 実行（take action）→ 検証（verify work）→ 繰り返し（repeat）」のフィードバックループとして定式化している。検証の手法として①明確なルールに基づく判定（最も効果的）②スクリーンショット等の視覚フィードバック③別モデルによる LLM ジャッジ、を挙げる。「ループを設計する」という発想自体は公式の設計思想であり、通称はそれを一つの技芸として名指したものである。

誰が言い出したか（提唱・定着の経緯）

2026年6月、複数の実務家がほぼ同時に同じ指摘をして定着した。Peter Steinberger が「エージェントにプロンプトを打ち込むのをやめ、エージェントを促すループを設計せよ（You should be designing loops that prompt your agents）」と要点を突き、Boris Cherny（Claude Code 作者）も同時期に同旨を発信、Addy Osmani が「loop engineering」と命名して呼び方が固まった。単発プロンプトの最適化（1インタラクション）に対し、複数インタラクションからなる系全体の最適化という切り分けである。

ループを構成する要素

要素設計する内容
① 起点（トリガー）人が毎回操作しなくても動き出す仕掛け（スケジュール・イベント・Stop hooks）
② ゴールと完了条件「コードを改善」ではなく「対象テストが全て通る」のように測れる形で事前定義する
③ コンテキスト・道具ループ内でエージェントが使う前提・ルール・許可された操作（＝ハーネス。§03 が前提になる）
④ 検証（判定役）テスト・型・リンター・CI 等の独立した判定。「モデルが良さそうと言う」は弱い検証
⑤ 状態の置き場進捗・気づき・約束事をどこに記録し引き継ぐか。最も見落とされがちで、自律稼働の成否を分ける
⑥ 停止条件完了・失敗・中断を何を証拠に判定し、止めた理由をどう次へ引き継ぐか（用語集「停止条件」）
⑦ 人のチェックポイント本番反映・顧客影響など、人の判断を残す箇所を先に決める
要の原則：判定役がボトルネック。ループには「作る側」と「できたかを判定する側」があり、作った本人（同じモデル・同じ文脈）に判定させない——自己採点は一貫して甘くなる。判定基準が甘いループは派手に壊れるのではなく、基準未満の成果物を自信たっぷりに量産し続ける。提唱側はこれを「エージェントが完了を証明できないなら、それはループエンジニアリングではなくドリフトの自動化だ」と言い切る（Operator Test）。力を注ぐ先は指示の言い回しでなく、「何をもって完了とするか」を測れる形にすること。

ゆるいループと厳格なループ／実装の重さ

  • ゆるいループ：目標と大まかな条件だけ与えて広く探索させる。発想は出るが、基準が曖昧だと消費だけがかさむ。
  • 厳格なループ：成功条件を事前固定し、毎回厳密に評価して明確に止める。安全だが、指定した以上のことは起きない。まず厳格で足場を固め、慣れてから探索の余地を持たせるのが安全な順序。
  • 実装の重さは段階的に：同じ指示の機械的反復（人が止める）→ 完了条件の軽い自動判定 → 進捗をファイルに記録しながら長時間動く常駐運用。自律度が高いほど良いのではなく、目的を達成できる最小限の重さを選ぶ。

Claude Code では、完了条件を設定して満たすまで自律継続する /goal、終了しようとした時に完了条件（テスト・カバレッジ・型チェック等)を検査して未達なら作業に差し戻す Stop hooks、定期実行の /loop・Routines などが、この営みの具体的な部品にあたる。

〈日常のたとえ〉

料理の「味見して、足りなければ調味料を足して、また味見する」という繰り返しに対し、「塩分濃度計でこの数値になったら仕上げる」と先に決めておくのがループエンジニアリング。基準がなければ気分次第でいつまでも調整が続くし、味見役を作った本人にやらせれば「もう十分おいしい」と甘く判定する。基準（完了条件）・計測器（判定役）・引き継ぎメモ（状態の置き場）を先に整えるから、任せて席を外せる。

使いどころ

「同じ依頼を何度も繰り返している」「夜間や不在時間にも進めたい」「完了判定が目視頼みになっている」が着手の合図。STEPUP ステップ5 の主戦場。順序は必ず「初回は人が確認 → 確認の積み重ねがルール → ルールが委譲を可能にする」で、観察が積み上がる前の自動化・ルール化を急がない。

05 三者の関係捨てるのではなく、外側に積む

3つの営みは選択肢ではなく入れ子である。プロンプトはハーネスの中で毎ターン渡される部品になり、ハーネスはループが安全に回るための受け入れ環境になる。

図2：入れ子の関係。ループはハーネスを前提にし、ハーネスはプロンプト（とモデル）を包む。外側の層が増えても内側の技芸は消えない。
「プロンプトはもう古い」ではない：システムプロンプトも CLAUDE.md も Skills もゴール定義も、中身はすべてプロンプトの技芸で書かれる。変わったのは「どこに書いて、どう届けるか」——都度チャットに書く（プロンプト）→ 環境に恒久配置する（ハーネス）→ 完了条件として機械に判定させる（ループ）。同様に、公式が一貫して要求する「成功基準を先に決めて実証的に測る」姿勢は3層すべてを貫く背骨であり、プロンプトの評価（eval）がそのままループの判定役に育っていく。

どこから手をつけるか（STEPUP-CLAUDE-01 との対応）

現在地（STEPUP の階段）主戦場になる営み次への移行の合図
ステップ1〜2.5（対話・成果物・外部化）プロンプトエンジニアリング毎回同じ前提・同じ工夫を書き直している
ステップ3〜3.5（エージェント・ハーネス）ハーネスエンジニアリング同じ失敗が再発する／「お願い」で縛ろうとしている
ステップ4〜5（マルチエージェント・ループ）ループエンジニアリング完了判定が目視頼み／不在時間に進めたい
順序を飛ばさない：ハーネス（環境と歯止め）が貧弱なままループを回すのは、ブレーキ未整備の車で自動運転を始めるに等しい。逆にハーネスが整っていれば、ループは「置き場の設計＋判定役の独立」という小さな追加で始められる（STEPUP §04 の運用例）。プロンプト→ハーネス→ループの順に、内側から積む。

06 事例：Rig3つのエンジニアリングを「ブリックで組む」実装

抽象論の締めくくりに、3層が1つの道具の中にどう同居するかを実例で確かめる。Rig（github.com/itoh-shun/rig）は Claude Code ネイティブ（command＋skill＋agents）のプラグインで、ブリック——persona・instruction・policy・knowledge／wiki・pattern・recipe・output-contract といった部品——を起動時に動的に組み合わせ、タスクに最適化されたエージェント・ハーネスをその場で engineering する汎用開発フロー・オーケストレータである。重い DSL エンジンや外部依存を持たず、ブリックを追加するだけで任意のフロー（開発3-Stage・PRレビュー・営業資料・合議判定など）が同じエンジンに乗る。名前も文字どおり「リグを組む（rig up）＝艤装する」——本資料の言葉でいえば、ハーネスエンジニアリングという営み自体を道具化したものにあたる。

3層との対応

層Rig での実装本資料の対応
プロンプト
（§02）		persona・instruction・output-contract はすべて散文 Markdown のブリックとして版管理される——「都度チャットに書く」工夫を、名前つきの再利用部品へ昇格させる。/rig:knowledge はドメイン知識を LLM-wiki（1概念＝1正準ページ・相互リンク）にし、persona へは事実を埋め込まず inject: [[slug]] で参照させる＝プロンプトの中身を暗黙知化させないプロンプトの技芸は消えず、恒久配置の部品になる（§05「捨てずに積む」）
ハーネス
（§03）		エンジンは PARSE → RESOLVE → COMPOSE → RUN。guides＝recipe・manifest（.claude/rig.md）・/rig:init が生成する CLAUDE.md 雛形。sensors＝reviewer 11枠（security／design／test 既定＋選択投入）・checks:（lint／test の機械検証）・acceptance-gate。そして強制側＝--orchestrate（遷移・ゲート・リトライ・停止を LLM でなくコードが強制する決定論ランナー）・worktree 隔離実行（run --isolate）・verifier の読み取り専用強制guides／sensors の二分法と「ガイダンス（お願い）と強制の分離」をそのまま部品化
ループ
（§04）		/rig:goal が高レベル目標を受け入れ基準に変換し、「現状把握→次手→既存フローへ委譲→照合」を達成まで周回する。判定役の独立＝実装と検証を別エージェントに分け、さらに finding-verifier が REJECT を証拠ベースで反証（誤検出の最終制御）。状態の置き場＝全 RUN が .rig/runs.jsonl に自動追記され、run-state は PreCompact フックでコンテキスト自動圧縮も生き延びるループ7要素（ゴール・判定役・状態・停止条件）の実装。判定役ボトルネック原則に対応
その先：判定役を「測る」（eval engineering の芽）：ループの質が判定役で決まるなら、次の問いは「その判定役は信用できるか」になる。Rig の /rig:drill は既知のバグの種を使い捨て worktree の差分に注入し、reviewer 別の検出／見逃し／誤検出スコアボードを実測する——判定役の品質を意見でなく数字にする仕組み。runs --personas は「5票以上 REJECT ゼロ＝ゴム印疑い」のような剪定ヒントを出す。用語集（GENAI-G01）が「ループエンジニアリングの次段」として挙げる eval engineering の実装例であり、なお Rig は本資料の主題である loop engineering の wiki ページを同梱して対応 reviewer に inject 済み——主題そのものが道具の中の部品として入っている、という入れ子になっている。

STEPUP-CLAUDE-01 の階段でいえば、Rig を導入することはステップ3.5（ハーネス）〜5（ループ）を一から自作せず「組む」ことに相当する。CLAUDE.md・権限・hooks を素手で積む代わりに、検証の独立・隔離実行・停止条件といった本資料の原則が既に部品化された足場から始められる。

読み方の注意：本節は「3層が実際にどう部品化されるか」を示す具体例としての紹介である。Rig は個人開発の OSS プラグインで、機能・コマンド・ブリック構成は時点で変わる。正本はリポジトリの README・skills/rig/SKILL.md（目録・flag・recipe スキーマ）で確認する。
